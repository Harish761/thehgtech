OpenAI Codex Security Finds 11,000 Bugs in 30 Days — 14 CVEs Assigned

OpenAI's AI security agent scanned 1.2 million commits across open-source projects, identified 792 critical and 10,561 high-severity vulnerabilities, and earned 14 CVEs — including stack buffer overflows in GnuPG and heap overflows in GnuTLS. Here's what it found, how it works, and what it means for application security teams.

What Is Codex Security?

On March 6, 2026, OpenAI released Codex Security — an AI-powered security agent designed to function as an automated application security engineer. Unlike traditional static analysis tools (SAST) that rely on pattern matching and rule sets, Codex Security uses agentic reasoning to analyze codebases, build project-specific threat models, validate potential vulnerabilities in a sandboxed environment, and propose actionable fixes.

The tool evolved from a private beta project codenamed "Aardvark" that launched in October 2025. During its initial 30-day testing period, Codex Security scanned 1.2 million code commits across various external open-source repositories.

The results were significant: 792 critical and 10,561 high-severity issues identified. Of those, 14 were serious enough to receive CVE identifiers from MITRE — meaning they represent real, exploitable vulnerabilities in widely-used open-source software.

Key Distinction: Codex Security is currently in research preview — available to ChatGPT Enterprise, Business, and Education customers. It is not a general-availability product. OpenAI has not published pricing or an enterprise SLA. Describing it as "production-ready" would be inaccurate as of this writing.

The 14 CVEs — What They Actually Found

The most compelling proof of Codex Security's capability isn't the volume of bugs — it's that 14 vulnerabilities were serious enough to be assigned CVE identifiers. These affect projects with millions of users worldwide:

CVE ID Project Vulnerability Type
CVE-2026-24881 GnuPG (gpg-agent) Stack buffer overflow
CVE-2026-24882 GnuPG (TPM2) Stack-based buffer overflow in PKDECRYPT
CVE-2025-32988 GnuTLS Double-free vulnerability
CVE-2025-32989 GnuTLS Heap-buffer overflow
CVE-2025-64175 GOGS 2FA bypass
CVE-2026-25242 GOGS Unauthenticated bypass
CVE-2025-35430 to CVE-2025-35436 Thorium Unauthenticated DoS, mail abuse, session management flaws

Why These CVEs Matter

GnuPG is the backbone of email encryption, package signing for Linux distributions, and SSH key management. A stack buffer overflow in gpg-agent (CVE-2026-24881) could potentially allow an attacker to execute arbitrary code during cryptographic operations. GnuPG is installed by default on nearly every Linux distribution and is used by package managers like apt and rpm to verify software signatures.

GnuTLS is a widely-used TLS library that underpins encrypted communications in hundreds of applications. A double-free vulnerability (CVE-2025-32988) and a heap-buffer overflow (CVE-2025-32989) in a TLS library are the types of bugs that, historically, have led to remote code execution exploits. For reference, the Heartbleed vulnerability (CVE-2014-0160) was a similar class of memory corruption in OpenSSL.

GOGS is a self-hosted Git service used by thousands of development teams. A 2FA bypass (CVE-2025-64175) means an attacker who has stolen a user's password can bypass the second authentication factor entirely — defeating the very purpose of MFA.

How Codex Security Works

According to OpenAI's announcement, Codex Security operates as an agentic AI security engineer that follows a multi-step workflow:

  1. Codebase Analysis: The agent ingests repository code, documentation, and commit history to build a context-aware understanding of the project
  2. Threat Modeling: It generates project-specific threat models based on the technology stack, attack surface, and data flows
  3. Vulnerability Discovery: Using its understanding of the codebase AND vulnerability patterns, it identifies potential issues — going beyond regex-matching to understand program semantics
  4. Sandbox Validation: Found vulnerabilities are validated in a sandboxed environment. This is the key differentiator — the AI tests whether the bug is actually exploitable, dramatically reducing false positives
  5. Fix Proposal: For confirmed vulnerabilities, it generates a proposed code fix with explanation
Why This Matters for False Positives: Traditional SAST tools like Semgrep, SonarQube, and Checkmarx are notorious for high false-positive rates (industry average: 30-50%). Codex Security's sandbox validation step — where it attempts to actually trigger the vulnerability — could address AppSec's biggest pain point. However, OpenAI has not published independent benchmarks comparing false-positive rates. This claim requires third-party verification.

Context: Is AI Better at Finding Bugs Than Humans?

The 11,353 bugs in 30 days is impressive on volume alone, but context matters:

What Makes This Different from Traditional Tools

  • Semantic understanding: Traditional SAST tools match patterns. Codex Security claims to understand what the code does, enabling it to find logic bugs that pattern-matching misses
  • Validation: The sandbox validation step is genuinely novel for an AI-first tool. Most AI code scanners report potential issues without confirming exploitability
  • Fix generation: Proposing fixes — not just finding bugs — addresses the remediation bottleneck that plagues security teams

What We Don't Know Yet

  • False-positive rate: How many of the 11,353 findings are actually exploitable? The 14 CVEs prove some are real, but what about the other 11,339?
  • False-negative rate: What bugs does it miss? No tool catches everything
  • Comparison to existing tools: How does this compare to running Semgrep + CodeQL + Snyk on the same repositories?
  • Code access: What happens to the code you submit? OpenAI's data handling policies for enterprise code are a significant concern
⚠️ Editorial Analysis: The "11,000 bugs" number is impressive but requires careful interpretation. Without knowing the false-positive rate, the denominator (total lines of code scanned), or how many of these were already known issues, the raw number is less meaningful than the 14 confirmed CVEs. The CVEs are the proof that matters — they demonstrate the tool can find real, novel vulnerabilities that human reviewers missed. The volume number is marketing; the CVEs are evidence.

Codex for OSS — Free Security for Open Source

Alongside the commercial tool, OpenAI announced "Codex for OSS" — a program offering free access to Codex Security tools and security review support for open-source maintainers. This is significant because:

  • Open-source projects often lack dedicated security resources
  • The 14 CVEs found by Codex were all in open-source projects
  • Critical infrastructure (GnuPG, GnuTLS) is often maintained by small teams or individuals with limited security budgets

If the program delivers, it could meaningfully improve the security posture of the open-source ecosystem. However, OpenAI has not yet published details on eligibility criteria, scan limits, or data retention policies for the free tier.

What Security Teams Should Do

RECOMMENDATIONS:
  1. If you're a ChatGPT Enterprise/Business customer: Request access to the Codex Security research preview and run it against a non-production codebase to evaluate its findings
  2. Check your GnuPG version: If you're running GnuPG < 2.5.x, check for patches addressing CVE-2026-24881 and CVE-2026-24882
  3. Check your GnuTLS version: Verify you're patched against CVE-2025-32988 (double-free) and CVE-2025-32989 (heap-buffer overflow)

  6. Open-source maintainers: Watch for OpenAI's "Codex for OSS" program details and apply for free access

Our Analysis: What This Means

Three takeaways from Codex Security's launch:

  1. AI vulnerability discovery is real. The 14 CVEs prove this isn't vaporware. These are real bugs in critical software that human reviewers missed. The GnuPG stack overflow, in particular, is the kind of vulnerability that security researchers spend months hunting for.
  2. The AppSec tool market is about to compress. If AI tools can find bugs faster and cheaper than traditional SAST tools, companies paying $100K+/year for Veracode, Checkmarx, or Fortify will demand answers about value. This doesn't mean traditional tools die overnight — but the pressure is real.
  3. Open-source security gets a boost — with caveats. The Codex for OSS program could genuinely help underfunded open-source projects. But it also creates a dependency on OpenAI's continued goodwill. If the free tier disappears or terms change, projects that relied on it will be exposed.

The most important question isn't whether AI can find bugs — Codex Security proved it can. The question is whether security teams can trust an AI's judgment enough to act on its findings without extensive human review. That trust will be built over time, with transparent benchmarks and independent testing. We're not there yet, but the 14 CVEs are a strong start.
