Operation PCPcat: 59,000 Next.js Servers Hacked in 48 Hours

A massive credential-theft campaign dubbed "PCPcat" compromised 59,128 Next.js and React servers in under 48 hours. But the real lesson isn't about patching—it's about why your "frontend" framework is now the most dangerous part of your infrastructure.

Attack: December 2025 Published: Dec 25, 2025 18 min read
URGENT - Mass Exploitation: Operation PCPcat achieved a 64.6% success rate across 91,505 scanned targets. If you run Next.js or React with server-side rendering (SSR), assume you were scanned. Check for CVE-2025-29927 and CVE-2025-66478 immediately.

The Uncomfortable Truth About "Frontend" Frameworks

Let me start with a controversial statement: Next.js and React are not frontend frameworks anymore. They haven't been for years. And this fundamental misunderstanding is why 59,128 servers just got pwned in 48 hours.

When developers think "React," they think browser. They think client-side rendering. They think "it's just JavaScript in the browser, what's the risk?" But modern React with Next.js is a full-stack Node.js server that:

  • Executes code on the server before sending HTML to the browser
  • Has direct access to environment variables containing production secrets
  • Can read and write to databases, APIs, and cloud infrastructure
  • Often runs with the same privileges as your backend services

Operation PCPcat didn't hack "websites." It hacked servers. And those servers had the keys to everything.

The Mental Model Problem: Security teams often treat Next.js deployments as "frontend" and apply weaker controls. Developers create `.env` files with production AWS keys because "it's just for the build." This cognitive gap between "React = browser" and "Next.js = server" is exactly what attackers exploit.

What Is Operation PCPcat?

Operation PCPcat is a large-scale, automated credential-theft campaign that targeted Next.js and React applications with server-side rendering capabilities. The campaign was named after the distinct cat-themed motifs found in the attack infrastructure.

59,128 Servers Compromised

64.6% Success Rate

The Attack Chain

The PCPcat operation leveraged two critical vulnerabilities in combination:

CVE-2025-29927 Critical

Next.js middleware authorization bypass allowing attackers to access protected routes and server-side functions without authentication.

CVE-2025-66478 Critical

Prototype pollution vulnerability in JSON payload processing enabling remote code execution via child_process.execSync().

How It Worked

  1. Mass Scanning: Attackers used automated scanners to identify public Next.js deployments across the internet
  2. Fingerprinting: Applications were fingerprinted to identify vulnerable versions
  3. Malware Delivery: The react.py malware was distributed to compromised hosts
  4. Prototype Pollution: Malicious JSON payloads exploited CVE-2025-66478 to achieve code execution
  5. RCE Confirmation: Attackers confirmed RCE with an id command test
  6. Credential Harvesting: Systematic extraction of secrets from compromised servers

What Was Stolen

The attackers systematically exfiltrated high-value credentials from every compromised server:

Target Location Impact
.env files Project root, /app, /src Database credentials, API keys, secrets
SSH Keys ~/.ssh/ Lateral movement to other servers
AWS Configs ~/.aws/credentials Full cloud infrastructure access
Docker Tokens ~/.docker/config.json Container registry access, supply chain risk
Git Credentials ~/.git-credentials, .gitconfig Source code access, supply chain attacks
Bash History ~/.bash_history Reconnaissance, additional secrets in commands

Technical Deep Dive: The react.py Malware

The react.py malware distributed during Operation PCPcat is a Python-based credential harvester designed specifically for server-side JavaScript environments. Key characteristics include:

Prototype Pollution Exploit

// Malicious JSON payload structure
{
  "__proto__": {
    "shell": "/bin/sh",
    "env": {
      "NODE_OPTIONS": "--require /tmp/malicious.js"
    }
  },
  "constructor": {
    "prototype": {
      "execSync": "function(cmd) { ... }"
    }
  }
}

The attack leverages JavaScript's prototype chain to inject malicious properties that are later accessed by the application, ultimately leading to command execution.

Why This Keeps Happening: The Framework Blindspot

1. The "Frontend" Security Gap

Most organizations apply different security standards to "frontend" vs "backend" systems. Next.js applications often:

  • Don't receive the same security audits as traditional backends
  • Are deployed by frontend developers unfamiliar with server security
  • Run in shared or less-secured infrastructure
  • Have lax secret management because "it's just the frontend"

2. The .env Credential Dump Problem

Next.js applications commonly store production secrets in .env files for convenience:

# Typical .env file in compromised Next.js apps
DATABASE_URL=postgresql://prod-user:SuperSecret123@prod-db.internal:5432/main
AWS_ACCESS_KEY_ID=AKIAIOSFODNN7EXAMPLE
AWS_SECRET_ACCESS_KEY=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
STRIPE_SECRET_KEY=sk_live_51...
GITHUB_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
SLACK_WEBHOOK_URL=https://hooks.slack.com/services/...

When attackers achieve RCE, this single file provides access to:

  • Production databases
  • Cloud infrastructure (AWS, GCP, Azure)
  • Payment processing (Stripe, etc.)
  • Internal communications (Slack)
  • Source code repositories (GitHub/GitLab tokens)

3. The "Build vs Runtime" Confusion

Developers often don't understand the difference between build-time and runtime environment variables in Next.js. Secrets meant only for builds end up accessible at runtime, expanding the attack surface.

"We put the AWS keys in .env.local because we thought they were only used during deployment. We didn't realize the running server could access them too." — Anonymous victim organization

Immediate Response Actions

If You Run Next.js or React SSR:
  1. Patch Immediately: Update Next.js to the latest patched version
  2. Assume Compromise: If you were running vulnerable versions, assume credentials were stolen
  3. Rotate ALL Secrets: Every credential in your .env files must be rotated
  4. Review AWS Activity: Check CloudTrail for unauthorized access using stolen keys
  5. Check SSH Keys: Audit authorized_keys on all connected servers
  6. Review Git History: Look for unauthorized commits or repository access

Checking Your Exposure

# Check Next.js version
cat package.json | grep next

# Search for sensitive files that may have been accessed
find /var/log -name "*.log" -exec grep -l "\.env\|\.ssh\|\.aws" {} \;

# Check for react.py or similar artifacts
find /tmp -name "*.py" -mtime -7 -ls

Secret Rotation Checklist

Secret Type Location Action
Database passwords .env, .env.local Rotate in DB, update .env
AWS Access Keys .env, ~/.aws/credentials Deactivate in IAM, create new keys
API Keys .env Regenerate in each service dashboard
SSH Keys ~/.ssh/ Generate new keypairs, remove old from authorized_keys
GitHub/GitLab Tokens .env, ~/.git-credentials Revoke in developer settings, create new PATs

Indicators of Compromise (IoCs)

Files to Look For

  • /tmp/react.py or variations
  • Unexplained Python scripts in temp directories
  • Modified .bashrc or .profile files
  • New cron jobs or scheduled tasks

Network Indicators

  • Outbound connections to unfamiliar IPs on ports 443, 8443, 9999
  • Large data transfers (credential dumps)
  • C2 beaconing patterns

SIEM Detection Query (Splunk)

index=web sourcetype=access_* 
| search (uri_path="*__proto__*" OR uri_path="*constructor*" OR uri_path="*.env*")
| stats count by src_ip, uri_path, status
| where count > 10

Long-Term Security Recommendations

1. Treat Next.js Like a Backend

Apply the same security controls you would to any Node.js server:

  • Regular security audits and penetration testing
  • Hardened container images
  • Network segmentation
  • Runtime application self-protection (RASP)

2. Migrate Away from .env Files

Use proper secret management solutions:

  • AWS Secrets Manager or HashiCorp Vault
  • Environment-specific injection rather than file-based storage
  • Runtime secret fetching instead of build-time embedding

3. Implement Least Privilege

Your Next.js application probably doesn't need:

  • Full AWS admin access
  • Write access to production databases
  • SSH keys to other infrastructure

4. Monitor for Framework Vulnerabilities

Subscribe to security advisories for:

  • Next.js: GitHub Security Advisories
  • React: Official security announcements
  • Node.js: Node.js Security Working Group
