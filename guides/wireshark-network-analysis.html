Wireshark Network Analysis

New

Master packet analysis for security investigations, troubleshooting, and network forensics.

20 min read
Beginner to Intermediate
Jan 17, 2026

Wireshark is the world's most popular network protocol analyzer. Whether you're troubleshooting connectivity issues, investigating malware communications, or performing network forensics, Wireshark is an essential tool in every security professional's toolkit.

In this comprehensive guide, I'll walk you through everything from basic packet capture to advanced filtering techniques used by SOC analysts to investigate security incidents.

Why Learn Wireshark?

Wireshark appears in nearly every SOC analyst job description. It's free, cross-platform, and provides unmatched visibility into network traffic. Understanding packet-level analysis is crucial for detecting lateral movement, exfiltration, and command-and-control communications.

1. Getting Started with Wireshark

Installation

Wireshark is available for Windows, macOS, and Linux:

# Windows/macOS
Download from https://www.wireshark.org/download.html

# Ubuntu/Debian
sudo apt update && sudo apt install wireshark

# Fedora/RHEL
sudo dnf install wireshark

# Add user to wireshark group (Linux)
sudo usermod -aG wireshark $USER

First Capture

To capture your first packets:

  1. Launch Wireshark
  2. Select your network interface (usually "eth0" or "Wi-Fi")
  3. Click the blue shark fin icon to start capturing
  4. Generate some traffic (browse a website)
  5. Click the red square to stop capturing
Quick Tip: Save captures in .pcapng format (Wireshark's native format) to preserve all metadata. Use .pcap when sharing with older tools.

2. Understanding the Interface

Wireshark's interface has three main panes:

Pane Location Purpose
Packet List Top Shows all captured packets with summary info (time, source, dest, protocol, info)
Packet Details Middle Expandable tree showing protocol layers (Ethernet → IP → TCP → HTTP)
Packet Bytes Bottom Raw hexadecimal and ASCII representation of selected packet

Color Coding

Wireshark uses colors to quickly identify traffic types:

  • Light Green - HTTP traffic
  • Light Blue - UDP traffic
  • Light Pink - TCP traffic
  • Red - Errors, retransmissions, bad checksums
  • Black on Yellow - ICMP traffic
  • Light Green - DNS traffic

3. Capture Filters (BPF Syntax)

Capture filters use Berkeley Packet Filter (BPF) syntax and filter packets during capture. This is useful when you know exactly what traffic you need and want to reduce file size.

Important: Capture filters permanently exclude non-matching packets. You cannot recover filtered-out traffic. When in doubt, capture everything and use display filters instead.

Common Capture Filters

# Capture only traffic to/from specific host
host 192.168.1.100

# Capture only traffic on specific port
port 443

# Capture only TCP traffic
tcp

# Capture traffic to specific network
net 10.0.0.0/8

# Combine with and/or
host 192.168.1.100 and port 80
tcp or udp

# Exclude specific traffic
not port 22

# Complex filter: HTTP but not from my IP
tcp port 80 and not host 192.168.1.50

4. Display Filters (Essential Reference)

Display filters use Wireshark's own syntax and filter the view of already-captured packets. This is the most common filtering method for analysis.

IP Address Filters

IP Filtering Examples
# Traffic involving specific IP
ip.addr == 192.168.1.100

# Source IP only
ip.src == 10.0.0.5

# Destination IP only
ip.dst == 8.8.8.8

# Traffic within a subnet
ip.addr == 192.168.1.0/24

# NOT from specific IP
!ip.addr == 192.168.1.1

Port and Protocol Filters

Port and Protocol Filtering
# Filter by TCP port
tcp.port == 443

# Filter by UDP port
udp.port == 53

# Source port only
tcp.srcport == 80

# Destination port only
tcp.dstport == 8080

# Port range (dynamic ports)
tcp.port >= 1024 && tcp.port <= 65535

# Protocol filters
http
dns
tls
icmp
arp

HTTP-Specific Filters

HTTP Traffic Analysis
# All HTTP traffic
http

# HTTP requests only
http.request

# HTTP responses only
http.response

# Specific HTTP method
http.request.method == "GET"
http.request.method == "POST"

# Filter by URL/URI
http.request.uri contains "/login"

# Filter by Host header
http.host contains "malicious-domain.com"

# Filter by User-Agent
http.user_agent contains "curl"

# HTTP response codes
http.response.code == 200
http.response.code == 404
http.response.code >= 400  # All errors

DNS Filters

DNS Analysis Filters
# All DNS traffic
dns

# DNS queries only
dns.flags.response == 0

# DNS responses only
dns.flags.response == 1

# Query for specific domain
dns.qry.name contains "suspicious.com"

# A record queries
dns.qry.type == 1

# TXT record queries (often used by malware)
dns.qry.type == 16

# NXDOMAIN responses (domain not found)
dns.flags.rcode == 3

TCP Flags and Connection Analysis

TCP Flag Filters
# SYN packets (connection attempts)
tcp.flags.syn == 1 && tcp.flags.ack == 0

# SYN-ACK (connection accepted)
tcp.flags.syn == 1 && tcp.flags.ack == 1

# FIN packets (connection close)
tcp.flags.fin == 1

# RST packets (connection reset/rejected)
tcp.flags.reset == 1

# Detect retransmissions (network issues)
tcp.analysis.retransmission

# Zero window (flow control issues)
tcp.analysis.zero_window

5. Protocol Analysis Techniques

Decrypting TLS/HTTPS Traffic

To analyze encrypted HTTPS traffic, you need either:

  1. Pre-master secret log file - Set SSLKEYLOGFILE environment variable in your browser
  2. Server's private key - For RSA key exchange (not ECDHE)
# Set in your shell before starting browser
export SSLKEYLOGFILE=~/sslkeys.log

# In Wireshark: Edit → Preferences → Protocols → TLS
# Set "(Pre)-Master-Secret log filename" to ~/sslkeys.log

SMB Analysis

# All SMB traffic
smb || smb2

# SMB file access
smb2.cmd == 5  # Create (file open)
smb2.cmd == 8  # Read
smb2.cmd == 9  # Write

# Filter by filename
smb2.filename contains "password"

6. Security Investigation Techniques

Detecting Port Scans

# Many SYN packets from single source (port scan indicator)
tcp.flags.syn == 1 && tcp.flags.ack == 0
# Then check Statistics → Endpoints → TCP tab

# RST responses indicate closed ports
tcp.flags.reset == 1

Detecting Data Exfiltration

# Large outbound transfers
ip.dst != 192.168.0.0/16 && tcp.len > 1000

# DNS tunneling (large TXT queries)
dns.qry.type == 16 && dns.qry.name.len > 50

# ICMP tunneling (large payloads)
icmp && data.len > 100

Detecting Beaconing

C2 beacons often show regular intervals. Use Statistics → IO Graph:

  1. Filter to suspect IP/port
  2. Set interval to 1 second
  3. Look for regular spikes (e.g., every 60 seconds)

7. Following TCP/HTTP Streams

One of Wireshark's most powerful features for security analysis:

  1. Right-click any packet in a conversation
  2. Select Follow → TCP Stream (or HTTP Stream)
  3. View the complete conversation reassembled
Pro Tip: Use "Follow Stream" to see full HTTP requests/responses, including POST data, headers, and responses. This reveals credentials sent in plaintext, file downloads, and API calls.

8. Statistics and Traffic Patterns

Essential Statistics Menus

Menu Use Case
Statistics → Endpoints Top talkers by IP, MAC, or port
Statistics → Conversations Traffic between specific IP pairs
Statistics → Protocol Hierarchy Protocol distribution breakdown
Statistics → IO Graph Traffic over time (detect beaconing)
Statistics → HTTP → Requests All HTTP requests by host/URI

9. Exporting Data and Objects

Export HTTP Objects (Files)

Extract files from HTTP traffic:

  1. File → Export Objects → HTTP
  2. View all downloaded files
  3. Save suspicious files for analysis

Export Other Formats

  • File → Export Specified Packets - Save filtered packets to new pcap
  • File → Export Packet Dissections - Export as CSV/JSON for SIEM
  • File → Export Objects → IMF - Extract email attachments

10. Malware Traffic Analysis

Common Malware Indicators

Indicator Filter Significance
Unusual User-Agent http.user_agent contains "python" or "curl" Scripted requests, not browsers
Long DNS queries dns.qry.name.len > 50 Possible DNS tunneling
Non-standard ports tcp.port == 4444 or tcp.port == 5555 Common RAT/backdoor ports
Base64 in URLs http.request.uri matches ".*[A-Za-z0-9+/]{20}.*" Encoded commands/data
Regular beaconing Use IO Graph C2 check-in intervals

11. Best Practices for SOC Analysts

SOC Analyst Checklist:
  • Always work on a copy of pcap files, never the original
  • Document your filters and findings as you go
  • Use time display format: View → Time Display Format → UTC Date and Time of Day
  • Create filter buttons for commonly used filters
  • Use coloring rules to highlight suspicious traffic
  • Check geolocation of external IPs (Statistics → Endpoints → Map)
  • Correlate timestamps with other log sources (SIEM, EDR)

Essential Filter Button Set

Create these as quick filter buttons (Analyze → Display Filter Expression → Save):

# C2 Hunt
http.user_agent contains "python" || http.user_agent contains "curl" || http.user_agent contains "wget"

# DNS Anomalies
dns.qry.type == 16 || dns.qry.name.len > 50

# Errors Only
tcp.analysis.flags || http.response.code >= 400

# Exclude noise
!(arp || stp || cdp || lldp)
