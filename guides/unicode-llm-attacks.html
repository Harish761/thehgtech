Unicode-Based LLM Attacks: Tokenization Exploitation via Invisible Characters

Technical analysis of attack vectors exploiting Unicode handling in language model tokenizers

25 min read | Intermediate | January 2026
Research Ethics & Responsible Disclosure: This document is published for defensive security research and educational purposes. All attack techniques described are derived from publicly disclosed academic research or vendor security advisories. No novel exploitation methods are introduced. Organizations are encouraged to use this material to assess and improve their LLM security posture. If you discover vulnerabilities in commercial systems, please follow responsible disclosure practices.
Abstract: This guide provides a technical analysis of Unicode-based attack vectors targeting large language models (LLMs). We examine how tokenizer behavior creates exploitable attack surfaces when processing invisible Unicode characters, emoji sequences, and tag code points. Drawing from peer-reviewed research (ICML 2025, arXiv 2024-2026) and disclosed vulnerabilities (CVE-2025-32711), we catalog demonstrated attack techniques, their experimental success conditions, and corresponding defense mechanisms. The material is intended for security researchers, red team practitioners, and organizations deploying LLM-integrated systems.

1. Attack Surface Overview

Unicode-based attacks against language models exploit a fundamental asymmetry: human perception of text differs from tokenizer processing. This asymmetry creates exploitable attack surfaces across the LLM processing pipeline.

Research Timeline (2024-2026)

The following research has established the viability of Unicode-based LLM attacks:

  • ICML 2025: Wei et al. demonstrated emoji insertion attacks achieving 15-75% bypass rates against safety classifiers, depending on target model and insertion strategy [1]
  • October 2025: "Imperceptible Jailbreaks" research showed variation selector sequences achieving 72-100% attack success rates in controlled experiments on open-source models [2]
  • May 2025: Microsoft disclosed CVE-2025-32711 ("EchoLeak")â€”ASCII smuggling via tag characters affecting Microsoft 365 Copilot [3]
  • January 2026: Follow-up research extended emoji-based techniques to commercial APIs with varying success [4]
Key Insight: These techniques exploit tokenizer behavior rather than model weights. This makes them transferable across model architectures sharing similar tokenizers (e.g., GPT-family BPE) while remaining sensitive to preprocessing pipelines.

3. Tokenization Mechanics

Understanding tokenizer behavior is essential to analyzing these attacks. Modern LLMs use subword tokenization (typically Byte-Pair Encoding or SentencePiece) which creates distinct tokens for Unicode code points.

The Perceptual-Processing Gap

Human Perception

Hello World

Renders as plain text; no anomalies visible

Tokenizer Processing

Hello[U+E0069][U+E0067][U+E006E]...World

Hidden code points encode additional tokens

The Tokenization Process

LLMs don't read text like humans - they break it into tokens. A token might be a word, part of a word, or even a single character. Critically, invisible Unicode characters get their own tokens.

Key Insight: The LLM's tokenizer processes every Unicode code point, including invisible ones. This creates a hidden channel for injecting instructions that bypass visual inspection.

Token Visualization Example

Normal text tokenization:

Hello World

Same text with hidden Unicode:

Hello [U+200B] [U+E0069] [U+E0067] [U+E006E] [U+E006F] [U+E0072] [U+E0065] World

The red tokens are invisible to humans but processed by the LLM as "ignore"

4. Emoji-Based Filter Bypass

Emoji insertion attacks exploit how Judge LLMs (safety classifiers) tokenize input differently than the target model. This technique was formalized in the ICML 2025 paper "Emoji Attack: A Method for Misleading Judge LLMs" [1].

Attack Mechanism

The attack operates by disrupting token boundaries in safety classifier input:

  1. Token fragmentation: Emojis inserted mid-word cause subword tokenizers to split words into smaller, semantically weaker tokens
  2. Pattern disruption: Safety classifiers trained on standard tokenization fail to recognize harmful patterns when token boundaries shift
  3. Semantic diffusion: The distributed tokens reduce the classifier's confidence in content categorization

What This Attack Does Not Break

  • Classifiers operating on normalized text (post-emoji stripping)
  • Multi-modal safety layers that analyze semantic intent separately
  • Systems with token-to-character ratio anomaly detection
  • Human review of flagged content (emojis remain visible)

5. Zero-Width Character Injection

Zero-width characters represent a class of Unicode code points that occupy no visual space during text rendering. These characters are tokenized by BPE-based algorithms, creating a channel for invisible instruction injection.

Zero-Width Character Reference

Character Name Code Point Legitimate Purpose Attack Use
Zero-Width Space U+200B Word breaking without visible space Token injection, separator
Zero-Width Non-Joiner U+200C Prevent character joining (Arabic/Persian) Binary encoding (bit 1)
Zero-Width Joiner U+200D Force character joining (emoji sequences) Binary encoding (bit 0)
Word Joiner U+2060 Prevent line breaks Token padding
Left-to-Right Mark U+200E Text direction control Rarely exploited directly
Right-to-Left Mark U+200F Text direction control Rarely exploited directly

Binary Encoding Technique

Attackers can encode arbitrary ASCII text using zero-width character pairs as a binary representation:

Encoding Schema:
U+200B (Zero-Width Space) = binary 0
U+200C (Zero-Width Non-Joiner) = binary 1

6. Tag Character Encoding

Unicode tag characters (U+E0000 to U+E007F) were originally designed for language tagging but are now deprecated. They remain in Unicode for compatibility - and attackers have found creative uses for them.

Tag Character Mapping

Each tag character corresponds to an ASCII character:

ASCII Tag Character Example
Space (32) U+E0020 Invisible space
A (65) U+E0041 Invisible "A"
a (97) U+E0061 Invisible "a"
i (105) U+E0069 Invisible "i"
Technical Note: Tag characters provide direct 1:1 ASCII mapping (offset 0xE0000), enabling full-text steganography without complex encoding. The string "ignore" encodes to 6 tag characters (U+E0069, U+E0067, U+E006E, U+E006F, U+E0072, U+E0065) â€” each invisible but tokenized by BPE.

Injection Pattern

Visible prompt: "Please summarize this article for me."

Tokenizer input (with hidden tag characters):
"Please summarize this article for me.[U+E0069][U+E0067][U+E006E][U+E006F][U+E0072][U+E0065]...(hidden instruction sequence)"

What This Attack Does Not Break

  • Systems that strip tag character range (U+E0000-U+E007F) at input
  • Tokenizers with restricted Unicode vocabularies (rare in production)
  • Byte-level inspection and logging (payload visible in hex dumps)

Attack Scenario Analysis

The following attack scenarios have been demonstrated in research or disclosed in security advisories:

Document Poisoning (Indirect Prompt Injection)

Vector: Attacker embeds invisible instructions in documents processed by RAG or document-QA systems.
Preconditions: Target system ingests external documents without Unicode sanitization.
Impact: Instruction execution, context hijacking, data exfiltration.

Email-Based Injection

Vector: Hidden instructions in email body processed by AI assistants (e.g., Microsoft Copilot pre-patch).
Preconditions: Email client/assistant preserves Unicode during ingestion.
Impact: Demonstrated in CVE-2025-32711: data exfiltration, unauthorized actions.

Clipboard/Paste Attacks

Vector: Malicious code snippets with hidden Unicode posted on forums/GitHub.
Preconditions: Developer pastes into AI coding assistant.
Impact: Hidden instruction execution in coding context.

8. Defense Implementation

For Developers Building AI Applications

Input Sanitization

Strip or replace suspicious Unicode characters before processing: 
# Python example: Remove invisible Unicode
import re

def sanitize_input(text):
    # Remove zero-width characters
    text = re.sub(r'[\u200B-\u200F\u2060-\u206F]', '', text)
    # Remove tag characters
    text = re.sub(r'[\uE0000-\uE007F]', '', text)
    # Remove variation selectors
    text = re.sub(r'[\uFE00-\uFE0F]', '', text)
    return text
Unicode Normalization

Normalize all input to a standard form: 
import unicodedata

def normalize_input(text):
    # NFKC normalization converts compatible characters
    return unicodedata.normalize('NFKC', text)

Defense-in-Depth Checklist

Layer Defense Purpose
Input Unicode sanitization Remove invisible characters
Input Character whitelisting Only allow expected character sets
Processing Tokenizer-aware validation Check token count vs character count
Output Response validation Verify output matches expected format
Monitoring Anomaly detection Flag unusual Unicode patterns

For End Users

Security Tips:
  • Be cautious when copy-pasting text from untrusted sources into AI assistants
  • Use text editors that can reveal hidden characters (like VS Code with "Render Whitespace")
  • If something seems off with AI responses, consider the input might be compromised
  • For sensitive AI workflows, type prompts manually instead of pasting

9. Detection Engineering

Python Detection Script

# Unicode Smuggling Detector
# Detects hidden/invisible characters in text

import re

def detect_unicode_smuggling(text):
    findings = []
    
    # Zero-width characters
    zwc = re.findall(r'[\u200B-\u200F\u2060-\u206F]', text)
    if zwc:
        findings.append(f"Zero-width characters: {len(zwc)} found")
    
    # Tag characters
    tags = re.findall(r'[\uE0000-\uE007F]', text)
    if tags:
        findings.append(f"Tag characters: {len(tags)} found")
    
    # Variation selectors
    vs = re.findall(r'[\uFE00-\uFE0F\U000E0100-\U000E01EF]', text)
    if vs:
        findings.append(f"Variation selectors: {len(vs)} found")
    
    # Bidirectional overrides
    bidi = re.findall(r'[\u202A-\u202E\u2066-\u2069]', text)
    if bidi:
        findings.append(f"Bidirectional overrides: {len(bidi)} found")
    
    # Token expansion check
    visible_chars = len(text.replace(' ', ''))
    if visible_chars > 0:
        ratio = len(text.encode('utf-8')) / visible_chars
        if ratio > 2:
            findings.append(f"Suspicious byte/char ratio: {ratio:.2f}")
    
    return findings if findings else ["No suspicious Unicode detected"]

10. Limitations & What This Guide Does Not Cover

This guide focuses on a specific class of attacks. The following limitations and out-of-scope topics should be noted:

Scope Limitations

  • Tokenizer-specific: Attack effectiveness varies significantly across tokenizer implementations. Results from BPE (GPT-family) may not transfer to SentencePiece or other tokenizers.
  • Model evolution: Commercial LLM providers continuously update safety mechanisms. Success rates reported in research may not reflect current production systems.
  • Preprocessing variability: Enterprise deployments often include custom preprocessing layers not present in research benchmarks.

Out-of-Scope Attack Classes

  • Gradient-based adversarial attacks (GCG): Covered in the advanced guide.
  • Multi-modal attacks: Image/audio-based prompt injection is not addressed here.
  • Social engineering jailbreaks: DAN prompts, roleplay exploits covered in LLM Jailbreaking Defense.
  • Agent-specific vulnerabilities: Tool-use exploitation, ReAct framework attacks.

Defense Limitations

  • False positive tradeoffs: Aggressive Unicode filtering may break legitimate multilingual content (Arabic ZWJ, CJK variation selectors).
  • Evasion vectors: Attackers may combine Unicode techniques with other methods to evade single-layer defenses.
  • No guarantee: Defense implementations described are mitigation strategies, not complete solutions.
