Active Directory Security Hardening

Defend the #1 enterprise attack target with tiered admin, Kerberos hardening, and attack detection strategies

28 min read | Intermediate | January 2026
Critical Target: Active Directory is compromised in over 80% of enterprise breaches. If attackers own AD, they own everything. This guide provides practical hardening steps you can implement today.

1. Why Active Directory is the Crown Jewel

Active Directory (AD) is the identity and access management backbone for 95% of Fortune 500 companies. When attackers compromise AD, they gain:

What They Get Impact
All user credentials Access to every account in the organization
Group memberships Ability to add themselves to any group
Computer accounts Lateral movement to any joined system
GPO control Push malware to every endpoint
Certificate authority access Issue certificates for any user
Think of it like this: Active Directory is like the master key system for a building. If someone steals the master key, they can open every door. AD security is not just an IT issue - it is a business survival issue.

The AD Attack Kill Chain

Attackers follow a predictable path when targeting AD:

  1. Initial Access - Phishing, vulnerability exploitation
  2. Credential Theft - Mimikatz, LSASS dumping
  3. Reconnaissance - BloodHound, AD enumeration
  4. Privilege Escalation - Kerberoasting, delegation abuse
  5. Lateral Movement - Pass-the-hash, pass-the-ticket
  6. Domain Dominance - DCSync, Golden Ticket

2. The Tiered Administrative Model

The tiered model is the foundation of AD security. It creates security boundaries that prevent attackers from moving from low-value to high-value targets.

Tier 0 - Domain Controllers and Identity

What belongs here:

  • Domain Controllers
  • AD FS, Azure AD Connect servers
  • PKI/Certificate Authority servers
  • Privileged Access Workstations (PAWs)
  • Domain Admin and Enterprise Admin accounts

Rule: Tier 0 admins NEVER log into Tier 1 or Tier 2 systems.

Tier 1 - Enterprise Servers

What belongs here:

  • Application servers
  • Database servers
  • File servers
  • Email servers
  • Server admin accounts

Rule: Tier 1 admins can manage servers but not workstations or DCs.

Tier 2 - Workstations and Users

What belongs here:

  • User workstations
  • Laptops
  • Help desk support
  • Workstation admin accounts

Rule: Tier 2 admins only manage end-user devices.

Implementing Tier Separation

# Create tiered admin groups
New-ADGroup -Name "Tier0-Admins" -GroupScope Global -Path "OU=Tier0,OU=Admin,DC=corp,DC=local"
New-ADGroup -Name "Tier1-Admins" -GroupScope Global -Path "OU=Tier1,OU=Admin,DC=corp,DC=local"
New-ADGroup -Name "Tier2-Admins" -GroupScope Global -Path "OU=Tier2,OU=Admin,DC=corp,DC=local"

# Use GPO to restrict logon
# Deny "Log on locally" for Tier0 admins on Tier1/Tier2 systems
# Deny "Log on through Remote Desktop Services" similarly

3. Securing Privileged Accounts

Password Best Practices

Account Type Minimum Length Rotation Storage
Domain Admin 25+ characters 90 days Emergency break-glass only
Service Accounts 25+ characters Use gMSA (automatic) Group Managed Service Accounts
Tier 0 Admin 20+ characters 90 days PAM solution
Tier 1/2 Admin 15+ characters 90 days Password manager

Protected Users Group

Add all privileged accounts to the Protected Users security group:

# Add admin to Protected Users
Add-ADGroupMember -Identity "Protected Users" -Members "AdminAccount"

# Verify membership
Get-ADGroupMember -Identity "Protected Users"

Protected Users provides:

  • No NTLM authentication (prevents pass-the-hash)
  • No credential caching
  • No DES or RC4 encryption in Kerberos
  • Cannot be delegated
Caution: Test Protected Users in a lab first. Some legacy applications may break when NTLM is disabled. Service accounts should use gMSAs instead of Protected Users.

LAPS for Local Admin Passwords

Local Administrator Password Solution (LAPS) ensures unique passwords on every machine:

# Deploy LAPS via GPO
# 1. Install LAPS management tools
Install-WindowsFeature RSAT-AD-Tools

# 2. Extend AD schema (one-time)
Update-AdmPwdADSchema

# 3. Set permissions
Set-AdmPwdComputerSelfPermission -OrgUnit "OU=Workstations,DC=corp,DC=local"

# 4. Allow IT to read passwords
Set-AdmPwdReadPasswordPermission -OrgUnit "OU=Workstations,DC=corp,DC=local" -AllowedPrincipals "Tier2-Admins"

4. Kerberos Hardening

Kerberos is the authentication protocol for AD. Hardening it prevents many common attacks.

Kerberoasting Defense

Kerberoasting extracts service account password hashes from TGS tickets. Attackers request tickets for SPNs, then crack them offline.
  • Use gMSAs - Automatic 240-character passwords that rotate automatically
  • Long passwords - 25+ characters for all service accounts
  • Monitor TGS requests - Alert on bulk ticket requests
  • Limit SPN accounts - Audit and remove unnecessary SPNs
# Find accounts with SPNs (potential Kerberoasting targets)
Get-ADUser -Filter {ServicePrincipalName -ne "$null"} -Properties ServicePrincipalName | 
    Select-Object Name, ServicePrincipalName

# Create a gMSA
New-ADServiceAccount -Name "svc-webapp" `
    -DNSHostName "svc-webapp.corp.local" `
    -PrincipalsAllowedToRetrieveManagedPassword "WebServers"

AS-REP Roasting Defense

AS-REP Roasting targets accounts with Kerberos pre-authentication disabled, allowing attackers to request encrypted data crackable offline.

Ensure NO accounts have "Do not require Kerberos preauthentication" enabled:

# Find vulnerable accounts
Get-ADUser -Filter {DoesNotRequirePreAuth -eq $True} -Properties DoesNotRequirePreAuth

# Fix vulnerable accounts
Set-ADAccountControl -Identity "VulnerableUser" -DoesNotRequirePreAuth $false

Disable Weak Encryption

# Via GPO: Computer Configuration > Policies > Windows Settings > 
# Security Settings > Local Policies > Security Options

# "Network security: Configure encryption types allowed for Kerberos"
# Enable ONLY: AES128_HMAC_SHA1, AES256_HMAC_SHA1
# Disable: DES_CBC_CRC, DES_CBC_MD5, RC4_HMAC_MD5

6. Detecting Common AD Attacks

DCSync Attack

DCSync abuses domain replication permissions to request password hashes directly from a DC. Attackers use Mimikatz to impersonate a domain controller.

Detection: Monitor Event ID 4662 for these GUIDs:

  • 1131f6aa-9c07-11d1-f79f-00c04fc2dcd2 - DS-Replication-Get-Changes
  • 1131f6ad-9c07-11d1-f79f-00c04fc2dcd2 - DS-Replication-Get-Changes-All

Prevention: Audit and restrict replication rights to only Domain Controllers.

Golden Ticket Attack

Golden Ticket uses the KRBTGT hash to forge TGTs for any user, including non-existent ones, with any group membership.
  • Rotate KRBTGT twice (with 10+ hour gap)
  • Monitor for TGT anomalies - Long lifetimes, impossible SIDs
  • Use Microsoft ATA/Defender for Identity
# Reset KRBTGT password (do this twice, 10+ hours apart)
# WARNING: May cause authentication issues - plan carefully
Reset-ADServiceAccountPassword -Identity krbtgt

# Verify last password change
Get-ADUser krbtgt -Properties PasswordLastSet | Select-Object PasswordLastSet

8. Quick Wins Checklist

Implement these changes today for immediate security improvement:

Week 1 Quick Wins

  • Add all Domain Admins to Protected Users group
  • Find and fix accounts with "Do not require Kerberos preauthentication"
  • Deploy LAPS to all workstations
  • Disable LM hash storage
  • Set LAN Manager to NTLMv2 only

Month 1 Goals

  • Implement tiered admin model
  • Convert service accounts to gMSAs
  • Enable advanced audit policies
  • Deploy Credential Guard on Windows 10/11
  • Set up alerting for critical Event IDs

Quarter 1 Goals

  • Deploy PAWs for Tier 0 administration
  • Implement just-in-time (JIT) admin access
  • Complete KRBTGT rotation
  • Restrict NTLM across the environment
  • Deploy Microsoft Defender for Identity or similar
