Sigma Rules for SIEM DetectionNEW

Write once, detect everywhere. Universal detection rules for Splunk, Elastic, Sentinel, and more.

18 min read Intermediate Jan 9, 2026

Sigma is the game-changer for detection engineering. Instead of writing the same detection logic in Splunk SPL, Elastic KQL, and Microsoft KQL, you write one Sigma rule and convert it to any SIEM. It's detection-as-code done right.

What is Sigma?

Sigma is an open standard for writing detection rules in a SIEM-agnostic YAML format. Rules can be converted to 40+ target platforms including Splunk, Elastic, Microsoft Sentinel, QRadar, Chronicle, and more.

1. Anatomy of a Sigma Rule

Basic Sigma Rule Structure
title: Suspicious PowerShell Download
id: 3b0047b3-5f8a-4c25-9c6b-6d5a96f9c8d7
status: experimental
description: Detects PowerShell downloading files from internet
references:
    - https://thehgtech.com/guides/threat-hunting-techniques
author: TheHGTech
date: 2026/01/09
tags:
    - attack.execution
    - attack.t1059.001
logsource:
    category: process_creation
    product: windows
detection:
    selection:
        Image|endswith: '\powershell.exe'
        CommandLine|contains|all:
            - 'http'
            - 'download'
    condition: selection
falsepositives:
    - Legitimate admin scripts
level: medium

Required Fields

Field Purpose Example
title Short descriptive name "Mimikatz Execution"
logsource Log type to search process_creation, windows
detection Matching logic selection + condition

Recommended Fields

Field Purpose
id UUID for tracking
status experimental, test, stable
tags MITRE ATT&CK mappings
level informational, low, medium, high, critical
falsepositives Known FP scenarios

2. Detection Logic

Selection Blocks

detection:
    selection_process:
        Image|endswith: '\cmd.exe'
    selection_args:
        CommandLine|contains:
            - '/c'
            - '/k'
    condition: selection_process and selection_args

Condition Operators

# AND - all must match
condition: selection1 and selection2

# OR - any can match  
condition: selection1 or selection2

# NOT - exclude matches
condition: selection and not filter

# Counting
condition: selection | count() > 5

# Aggregation
condition: selection | count(User) by Computer > 10

4. Field Modifiers

Modifier Description Example
contains Substring match CommandLine|contains: 'mimikatz'
startswith Prefix match Image|startswith: 'C:\Temp'
endswith Suffix match Image|endswith: '.exe'
all All values must match CommandLine|contains|all:
base64 Base64 encoded CommandLine|base64offset|contains:
re Regex pattern CommandLine|re: '\\d{1,3}\\.\\d+'
cidr IP range DestinationIp|cidr: '10.0.0.0/8'

5. Real-World Examples

Cobalt Strike Beacon Detection
title: Cobalt Strike Named Pipe
id: d5601f8c-b76a-4f4e-bca1-4a3c55645a62
status: stable
description: Detects Cobalt Strike default named pipes
author: TheHGTech
tags:
    - attack.execution
    - attack.t1059
logsource:
    category: pipe_created
    product: windows
detection:
    selection:
        PipeName|startswith:
            - '\MSSE-'
            - '\postex_'
            - '\status_'
            - '\msagent_'
    condition: selection
level: critical
Credential Dumping - LSASS Access
title: LSASS Memory Access
id: 32d0d3e2-e58d-4d41-926b-18b520b2b32d
status: stable
description: Detects process accessing LSASS memory
author: TheHGTech
tags:
    - attack.credential_access
    - attack.t1003.001
logsource:
    category: process_access
    product: windows
detection:
    selection:
        TargetImage|endswith: '\lsass.exe'
        GrantedAccess|contains:
            - '0x1010'
            - '0x1410'
            - '0x1438'
    filter:
        SourceImage|endswith:
            - '\wmiprvse.exe'
            - '\taskmgr.exe'
    condition: selection and not filter
level: high
Suspicious Scheduled Task Creation
title: Suspicious Scheduled Task
id: 7fb9a7ce-8c6e-4a84-b6d0-f8b8c6e9c8d7
status: experimental
description: Detects scheduled task creation via command line
author: TheHGTech
tags:
    - attack.persistence
    - attack.t1053.005
logsource:
    category: process_creation
    product: windows
detection:
    selection_cmd:
        Image|endswith:
            - '\schtasks.exe'
            - '\at.exe'
        CommandLine|contains: '/create'
    selection_suspicious:
        CommandLine|contains:
            - 'powershell'
            - 'cmd /c'
            - 'http'
            - 'ftp'
    condition: selection_cmd and selection_suspicious
level: medium

6. Converting Sigma Rules

Using sigma-cli

# Install
pip install sigma-cli

# Convert to Splunk
sigma convert -t splunk -p sysmon rule.yml

# Convert to Elastic
sigma convert -t elasticsearch -p ecs_windows rule.yml

# Convert to Microsoft Sentinel
sigma convert -t microsoft365defender rule.yml

# Batch convert directory
sigma convert -t splunk -p sysmon rules/ -o output/

Output Examples

Splunk SPL:

index=windows EventCode=1 
| where match(Image, "(?i)\\\\powershell\\.exe$")
| where match(CommandLine, "(?i).*http.*download.*")

Elastic KQL:

process.name:powershell.exe AND 
process.command_line:(*http* AND *download*)

7. Detection Engineering Workflow

  1. Intelligence: Gather threat intel on TTPs to detect
  2. Write Rule: Create Sigma rule with proper log source
  3. Test: Validate against sample data locally
  4. Convert: Generate SIEM-specific queries
  5. Deploy: Add to detection pipeline
  6. Tune: Reduce false positives based on alerts
  7. Version: Track changes in Git
Best Practices:
  • Always include MITRE ATT&CK tags
  • Document false positives you've encountered
  • Use specific log sources, not generic
  • Test rules before production deployment
  • Version control all rules in Git
  • Include references to threat intel sources
