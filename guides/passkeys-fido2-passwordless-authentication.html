Passkeys & FIDO2: The Enterprise Passwordless Authentication Guide

WebAuthn registration/authentication flows, attestation deep dive, synced vs device-bound passkeys, AiTM bypass defense, and production deployment patterns

28 min read | Intermediate to Advanced | February 2026
Research Ethics & Responsible Disclosure: This document is published for defensive security research and educational purposes. Attack scenarios described are based on publicly disclosed research. Organizations are encouraged to use this material to assess and improve their authentication security posture.
Abstract: Passwords are the #1 attack vector in 2026 — responsible for 80%+ of breaches via credential stuffing, phishing, and brute force. FIDO2 passkeys eliminate this attack surface entirely through asymmetric cryptography bound to device hardware. This guide provides a deep technical walkthrough of the WebAuthn registration and authentication protocols, attestation mechanisms, the security tradeoffs between synced and device-bound passkeys, real-world MFA bypass attacks (AiTM, push fatigue, SIM swap) and how passkeys defeat them, plus production deployment patterns for enterprise environments using Entra ID, Okta, and custom WebAuthn implementations.

1. Threat Model: Why Passwords Must Die

2. FIDO2/WebAuthn Architecture Deep Dive

2.1 Protocol Stack

FIDO2 is not a single protocol — it's a stack of three interlocking specifications:

┌─────────────────────────────────────────────────────────────────┐
│                     FIDO2 PROTOCOL STACK                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  LAYER 3: WebAuthn API (W3C Standard)                    │    │
│  │  navigator.credentials.create() / .get()                 │    │
│  │  Browser ↔ Relying Party (your server)                   │    │
│  └──────────────────────┬──────────────────────────────────┘    │
│                         │                                        │
│  ┌──────────────────────┴──────────────────────────────────┐    │
│  │  LAYER 2: CTAP2 (Client to Authenticator Protocol)       │    │
│  │  Browser/OS ↔ Authenticator (USB/NFC/BLE/Platform)       │    │
│  │  Handles: PIN/Biometric verification, key generation     │    │
│  └──────────────────────┬──────────────────────────────────┘    │
│                         │                                        │
│  ┌──────────────────────┴──────────────────────────────────┐    │
│  │  LAYER 1: Authenticator Hardware                         │    │
│  │  TPM 2.0 | Secure Enclave | Security Key (YubiKey)      │    │
│  │  Private key NEVER leaves this boundary                  │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  KEY PRINCIPLE: Asymmetric cryptography                          │
│  Server stores: Public Key (useless if stolen)                   │
│  Device stores: Private Key (in hardware, non-exportable)        │
│  Authentication: Server sends challenge → Device signs with      │
│  private key → Server verifies with public key                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

Server-Side Registration (Node.js + @simplewebauthn/server)

import {
  generateRegistrationOptions,
  verifyRegistrationResponse
} from '@simplewebauthn/server';

// Step 1: Generate registration options
const options = await generateRegistrationOptions({
  rpName: 'MyApp',
  rpID: 'myapp.com',
  userID: user.id,               // Opaque user handle (NOT email)
  userName: user.email,
  attestationType: 'direct',     // Request attestation for enterprise
  authenticatorSelection: {
    residentKey: 'required',     // Discoverable credential (passkey)
    userVerification: 'required', // Require biometric/PIN
    authenticatorAttachment: 'platform', // Or 'cross-platform' for keys
  },
  supportedAlgorithmIDs: [-7, -257], // ES256 (P-256), RS256
});

// Store challenge in session for verification
session.currentChallenge = options.challenge;

// Step 2: Verify the registration response from client
const verification = await verifyRegistrationResponse({
  response: clientResponse,        // From navigator.credentials.create()
  expectedChallenge: session.currentChallenge,
  expectedOrigin: 'https://myapp.com',
  expectedRPID: 'myapp.com',
  requireUserVerification: true,
});

if (verification.verified) {
  // Store credential in database
  await db.credentials.create({
    credentialID: verification.registrationInfo.credentialID,
    publicKey: verification.registrationInfo.credentialPublicKey,
    counter: verification.registrationInfo.counter,
    transports: clientResponse.response.getTransports(),
    userID: user.id,
    // AAGUID for authenticator identification
    aaguid: verification.registrationInfo.aaguid,
  });
}

     │                      │                      │  - Lookup public key
     │                      │                      │  - Verify signature
     │                      │                      │  - Check counter (clone)

     │                      │◀─────────────────────│

KEY SECURITY PROPERTIES:
→ Challenge is UNIQUE per attempt (prevents replay)
→ RP ID is checked by authenticator (prevents phishing)
→ Private key NEVER transmitted (prevents interception)
→ Counter increments (detects cloned authenticators)

5. Attestation: Verifying Authenticator Trust

Attestation answers a critical enterprise question: "Can I trust the device that generated this passkey?"

Attestation TypeWhat Server ReceivesUse Case
NoneNo attestation dataConsumer apps — maximum privacy, minimum trust
IndirectAnonymized attestation via privacy CABalance between privacy and device verification
DirectFull attestation certificate chain + AAGUIDEnterprise — verify exact authenticator make/model
EnterpriseDevice-specific serial number + certsHigh-assurance: government, military, finance

Verifying Attestation Against FIDO Metadata Service (MDS)

// Enterprise: Verify the authenticator is on your approved list
import { MetadataService } from '@simplewebauthn/server';

// Initialize with FIDO Alliance Metadata Service
await MetadataService.initialize({
  mdsServers: [{ url: 'https://mds3.fidoalliance.org/' }],
});

// After registration, check the AAGUID
const aaguid = verification.registrationInfo.aaguid;
const metadata = await MetadataService.getStatement(aaguid);

if (!metadata) {
  throw new Error('Unknown authenticator — not in FIDO MDS');
}

// Enforce enterprise policy: only allow FIPS 140-2 certified keys
const certifications = metadata.authenticatorGetInfo?.certifications;
if (!certifications?.FIPS_140_2) {
  throw new Error('Authenticator not FIPS 140-2 certified — rejected');
}

// Log the authenticator model for inventory
console.log(`Registered: ${metadata.description}`);
// → "Registered: YubiKey 5 Series (FIPS)"

6. Synced vs Device-Bound Passkeys

PropertySynced PasskeysDevice-Bound Passkeys
StorageiCloud Keychain, Google PM, 1PasswordYubiKey, Titan Key, TPM only
Cross-Device✅ Available on all synced devices❌ Only on the hardware key
Backup/Recovery✅ Cloud backup (encrypted)❌ Lost key = register new one
ExportableWithin ecosystem only❌ Non-exportable by design
Cloud DependencyRequires cloud account securityNone — fully offline capable
Phishing Resistant✅ Yes — origin bound✅ Yes — origin bound
Enterprise Attestation❌ Limited✅ Full (AAGUID, certs, FIPS)
Best ForConsumer apps, low-medium risk enterpriseFinance, healthcare, government, privileged access
Security Consideration — Synced Passkey Risks:

Synced passkeys are only as secure as the cloud account protecting them. If an attacker compromises an Apple ID or Google account (via social engineering, SIM swap, or account recovery abuse), they gain access to ALL synced passkeys. For privileged accounts (admin, root, financial), mandate device-bound passkeys on FIPS hardware.

7. MFA Bypass Attacks & How Passkeys Defeat Them

7.1 Adversary-in-the-Middle (AiTM) — Evilginx

How It Works: Attacker deploys a reverse proxy (Evilginx, Modlishka) that sits between the victim and the real login page. The victim sees what looks like the real site. They enter their password and MFA code. The proxy captures both and replays them to the real server, stealing the session cookie.

Why Passwords + TOTP Fail: Both the password AND the TOTP code are transmitted through the proxy to the real server. The attacker captures the resulting session token.

Why Passkeys Win: The authenticator checks the RP ID (domain) before signing. The proxy domain (login-microsoft.evil.com) doesn't match the RP ID (login.microsoft.com), so the authenticator refuses to sign the challenge. Attack fails silently.

7.2 Push Fatigue (MFA Bombing)

How It Works: Attacker has stolen credentials. They trigger MFA push notifications repeatedly (often at 2-3 AM) until the exhausted user approves one.

Victims: Uber (Lapsus$, 2022), Cisco (Yanluowang, 2022), Microsoft (Midnight Blizzard, 2024)

Why Passkeys Win: There is no push notification to spam. Authentication requires biometric verification on an origin-bound credential. The attacker can't even initiate the ceremony without physical access to the authenticator.

7.3 SIM Swap + SMS OTP Interception

How It Works: Attacker social-engineers the mobile carrier to port the victim's phone number to a SIM they control. All SMS-based OTPs now go to the attacker.

Why Passkeys Win: No SMS channel involved. Authentication is purely cryptographic — the private key lives in the device's TPM/Secure Enclave, not on a SIM card.

9. WebAuthn Client-Side Implementation

Complete Client-Side WebAuthn (Browser JavaScript)

// ===== REGISTRATION =====
async function registerPasskey() {
  // 1. Get options from your server
  const optionsRes = await fetch('/api/webauthn/register/begin', {
    method: 'POST',
    credentials: 'same-origin',
  });
  const options = await optionsRes.json();

  // 2. Decode base64url fields
  options.challenge = base64urlToBuffer(options.challenge);
  options.user.id = base64urlToBuffer(options.user.id);

  // 3. Create credential — triggers biometric prompt
  const credential = await navigator.credentials.create({
    publicKey: options
  });

  // 4. Send attestation to server for verification
  const result = await fetch('/api/webauthn/register/finish', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      id: credential.id,
      rawId: bufferToBase64url(credential.rawId),
      type: credential.type,
      response: {
        attestationObject: bufferToBase64url(
          credential.response.attestationObject
        ),
        clientDataJSON: bufferToBase64url(
          credential.response.clientDataJSON
        ),
      },
    }),
  });
  return result.json();
}

// ===== AUTHENTICATION =====
async function loginWithPasskey() {
  // 1. Get authentication options
  const optionsRes = await fetch('/api/webauthn/auth/begin', {
    method: 'POST',
    credentials: 'same-origin',
  });
  const options = await optionsRes.json();
  options.challenge = base64urlToBuffer(options.challenge);

  // 2. Get assertion — triggers biometric prompt
  const assertion = await navigator.credentials.get({
    publicKey: options
  });

  // 3. Send signed assertion to server
  const result = await fetch('/api/webauthn/auth/finish', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      id: assertion.id,
      rawId: bufferToBase64url(assertion.rawId),
      type: assertion.type,
      response: {
        authenticatorData: bufferToBase64url(
          assertion.response.authenticatorData
        ),
        clientDataJSON: bufferToBase64url(
          assertion.response.clientDataJSON
        ),
        signature: bufferToBase64url(
          assertion.response.signature
        ),
        userHandle: bufferToBase64url(
          assertion.response.userHandle
        ),
      },
    }),
  });
  return result.json();
}

10.2 Audit Checklist

Passkey Deployment Audit Script: 
#!/bin/bash
# passkey-audit.sh — Verify FIDO2 deployment hardening

echo "=== FIDO2/Passkey Deployment Audit ==="

# 1. Check WebAuthn RP ID configuration
echo "[CHECK] RP ID matches production domain..."
RPID=$(grep -r "rpID" /app/config/ 2>/dev/null | head -1)
echo "  RP ID: $RPID"

# 2. Verify user verification is REQUIRED (not 'preferred')
echo "[CHECK] User verification enforcement..."
UV=$(grep -r "userVerification" /app/config/ 2>/dev/null)
if echo "$UV" | grep -q "required"; then
    echo "  ✅ UV = required"
else
    echo "  ❌ WARNING: UV not set to 'required' — biometric bypass possible"
fi

# 3. Check attestation mode
echo "[CHECK] Attestation type..."
ATT=$(grep -r "attestationType\|attestation" /app/config/ 2>/dev/null)
echo "  $ATT"

# 4. Verify HTTPS enforcement
echo "[CHECK] Origin uses HTTPS..."
ORIGIN=$(grep -r "expectedOrigin" /app/config/ 2>/dev/null)
if echo "$ORIGIN" | grep -q "https://"; then
    echo "  ✅ HTTPS enforced"
else
    echo "  ❌ CRITICAL: HTTP origin detected — WebAuthn requires HTTPS"
fi

# 5. Check supported algorithms
echo "[CHECK] Algorithm support..."
ALGOS=$(grep -r "supportedAlgorithmIDs\|pubKeyCredParams" /app/config/ 2>/dev/null)
echo "  $ALGOS"
echo ""
echo "=== Audit Complete ==="
