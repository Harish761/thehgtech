AWS Security Best Practices

Cloud

Secure your AWS environment from day one with comprehensive hardening guidelines.

25 min read
Intermediate
Jan 17, 2026

AWS powers millions of organizations worldwide, making it a prime target for attackers. Misconfigurations are the #1 cause of cloud breaches - from exposed S3 buckets to overly permissive IAM roles. This guide covers the essential security controls every AWS environment needs.

The AWS Shared Responsibility Model

AWS is responsible for: Physical security, hypervisor, network infrastructure
You are responsible for: IAM, data encryption, network configuration, application security, OS patching

1. Account Security Fundamentals

Root Account Protection

Critical: Secure Your Root Account

The root account has unlimited access to everything. Compromise of root means total account takeover.

Root Account Checklist
  • Enable MFA (hardware key preferred, TOTP minimum)
  • Never create access keys for root
  • Use a dedicated email with strong security
  • Only use root for tasks that require it
  • Set up alternate contacts for billing/security

AWS Organizations

For multi-account environments, use AWS Organizations:

  • Service Control Policies (SCPs) - Restrict what services accounts can use
  • Consolidated billing - Single payer, multiple accounts
  • Account isolation - Separate prod/dev/security accounts

2. IAM Best Practices

The Principle of Least Privilege

Grant only the minimum permissions required for a task:

Good vs Bad IAM Policy
// BAD - Too permissive
{
    "Effect": "Allow",
    "Action": "s3:*",
    "Resource": "*"
}

// GOOD - Specific actions and resources
{
    "Effect": "Allow",
    "Action": [
        "s3:GetObject",
        "s3:ListBucket"
    ],
    "Resource": [
        "arn:aws:s3:::my-app-bucket",
        "arn:aws:s3:::my-app-bucket/*"
    ]
}

IAM Roles vs Access Keys

Use Case Solution Why
EC2 accessing S3 IAM Role (Instance Profile) No keys to manage or rotate
Lambda functions IAM Role (Execution Role) Automatic, secure
Cross-account access IAM Role with AssumeRole No shared credentials
CI/CD pipelines OIDC Federation No long-lived keys
Local development SSO / Identity Center Temporary credentials

IAM Access Analyzer

Enable IAM Access Analyzer to find:

  • Resources shared with external accounts
  • Unused permissions (for policy trimming)
  • Public or cross-account access paths
# AWS CLI - Create analyzer
aws accessanalyzer create-analyzer \
    --analyzer-name my-analyzer \
    --type ACCOUNT

3. S3 Bucket Security

S3 Misconfigurations = Data Breaches

Public S3 buckets have caused countless data leaks. Always assume your buckets contain sensitive data.

S3 Block Public Access

Enable at the account level to prevent any bucket from becoming public:

# Enable Block Public Access for entire account
aws s3control put-public-access-block \
    --account-id 123456789012 \
    --public-access-block-configuration \
        "BlockPublicAcls=true,\
         IgnorePublicAcls=true,\
         BlockPublicPolicy=true,\
         RestrictPublicBuckets=true"

S3 Security Checklist

Every S3 Bucket Should Have:
  • Block Public Access enabled
  • Default encryption (SSE-S3 or SSE-KMS)
  • Versioning enabled
  • Access logging enabled
  • Lifecycle policies for old versions
  • Object Lock for compliance data (if required)

Secure Bucket Policy Example

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "DenyInsecureTransport",
            "Effect": "Deny",
            "Principal": "*",
            "Action": "s3:*",
            "Resource": [
                "arn:aws:s3:::my-bucket",
                "arn:aws:s3:::my-bucket/*"
            ],
            "Condition": {
                "Bool": {"aws:SecureTransport": "false"}
            }
        }
    ]
}

4. VPC and Network Security

Security Groups Best Practices

Rule Bad Practice Good Practice
Inbound SSH 0.0.0.0/0:22 VPN/Bastion IP only
Inbound RDP 0.0.0.0/0:3389 Never public; use SSM
Database All IPs:3306 App security group only
Outbound All traffic allowed Restrict to needed destinations

VPC Flow Logs

Enable VPC Flow Logs for network visibility:

aws ec2 create-flow-logs \
    --resource-type VPC \
    --resource-ids vpc-12345678 \
    --traffic-type ALL \
    --log-destination-type cloud-watch-logs \
    --log-group-name /aws/vpc/flowlogs

Use VPC Endpoints

Keep traffic private with VPC Endpoints:

  • Gateway Endpoints - S3, DynamoDB (free)
  • Interface Endpoints - Other AWS services (paid)

5. Logging and Monitoring

CloudTrail - The Foundation

CloudTrail logs all API activity. Enable it in all regions:

CloudTrail Configuration
  • Create organization trail (if using Organizations)
  • Enable in all regions
  • Log to S3 bucket with versioning + encryption
  • Enable log file integrity validation
  • Send to CloudWatch Logs for alerting
  • Protect trail from tampering with SCPs

CloudWatch Alarms

Set up alarms for critical security events:

  • Root account usage
  • IAM policy changes
  • Security group changes
  • CloudTrail disabled
  • Console login without MFA

6. Data Encryption

Encryption at Rest

Service Default Encryption Recommended
S3 Optional SSE-S3 or SSE-KMS
EBS Optional (can enforce) Enable account-level default
RDS Optional Enable at creation
DynamoDB SSE enabled by default Use KMS CMK for more control

KMS Key Policies

Use AWS KMS for key management. Control access with key policies:

{
    "Sid": "Allow use of the key",
    "Effect": "Allow",
    "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/MyAppRole"
    },
    "Action": [
        "kms:Encrypt",
        "kms:Decrypt",
        "kms:GenerateDataKey"
    ],
    "Resource": "*"
}

7. Threat Detection

Amazon GuardDuty

GuardDuty is AWS's threat detection service. Enable it in all accounts and regions:

  • Analyzes CloudTrail, VPC Flow Logs, DNS logs
  • Detects crypto mining, data exfiltration, backdoors
  • Machine learning identifies anomalies
  • Integrates with Security Hub

Security Hub

Centralize security findings from multiple services:

  • GuardDuty findings
  • Inspector findings
  • IAM Access Analyzer
  • Config Rules compliance
  • Third-party tools

8. Compliance and Auditing

AWS Config Rules

Continuously monitor configuration compliance:

# Essential Config Rules
- s3-bucket-public-read-prohibited
- s3-bucket-ssl-requests-only
- iam-root-access-key-check
- iam-user-mfa-enabled
- ec2-security-group-attached-to-eni
- vpc-flow-logs-enabled
- cloudtrail-enabled

AWS Audit Manager

Automate evidence collection for:

  • SOC 2
  • ISO 27001
  • PCI DSS
  • HIPAA
  • GDPR
