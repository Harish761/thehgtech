OAuth Security in 2026: How Attackers Exploit the Protocol You Trust

From trusted authorization standard to attack vector — device code phishing, redirection abuse, consent phishing, token persistence, and how to defend your organization

20 min read | Intermediate to Advanced | March 2026
Research Ethics & Responsible Disclosure: This guide is published for defensive security research and educational purposes. Attack techniques described are based on publicly disclosed research from Microsoft Threat Intelligence, CISA advisories, and peer-reviewed security publications. Organizations are encouraged to use this material to assess and improve their OAuth security posture.
Abstract: OAuth 2.0 has been the backbone of delegated authorization on the internet for over a decade. In 2025-2026, threat actors including Storm-2372 and operators of Phishing-as-a-Service platforms like Tycoon 2FA are exploiting legitimate OAuth features — device code flow, error-triggered redirections, and consent pages — to bypass MFA and gain persistent access to Microsoft 365 and Google Workspace accounts. This guide explains how OAuth works, why it was considered secure, what has changed, and provides verified defensive measures including Conditional Access policies, KQL detection queries, and app governance strategies.

1. What Is OAuth 2.0 and Why Was It Trusted?

1.1 The Problem OAuth Solved

Before OAuth, if a third-party application needed to access your data on another service (e.g., a calendar app reading your Gmail), you had to hand over your actual username and password. This was fundamentally broken — the third party had full access, couldn't be selectively revoked, and if compromised, exposed your entire account.

OAuth 2.0, defined in RFC 6749 (published October 2012), solved this by introducing delegated authorization. Instead of sharing passwords, users grant specific, scoped permissions to applications via tokens. The key principles were:

  • No password sharing — The third-party app never sees your password
  • Scoped access — Apps only get the permissions they request (e.g., "read email" but not "send email")
  • Revocable tokens — Users can revoke access without changing their password
  • Time-limited — Access tokens expire, requiring refresh

1.2 Why OAuth Was Considered Secure

For over a decade, OAuth 2.0 was widely regarded as a secure authorization framework because:

Security Property How OAuth Achieves It
Credential Isolation User credentials are only entered on the Identity Provider's domain (e.g., login.microsoftonline.com), never on the third-party app
Least Privilege Scopes limit what an app can access — users see exactly what permissions are requested
Token Expiration Access tokens are short-lived (typically 1 hour for Microsoft Entra ID), limiting the window of compromise
Client Authentication Confidential clients must authenticate with client secrets or certificates at the token endpoint
Redirect URI Validation Authorization servers validate redirect URIs against pre-registered values, preventing code interception
PKCE (RFC 7636) Proof Key for Code Exchange prevents authorization code interception in public clients
Key Insight:

The OAuth 2.0 specification itself is not broken. What attackers are exploiting in 2025-2026 are legitimate, by-design features of the protocol — the same redirect behavior, device code flow, and consent mechanisms that make OAuth flexible are being weaponized. This is what makes these attacks so difficult to detect — they use trusted infrastructure.

3. What Changed: The 2025-2026 Attack Landscape

So if OAuth itself isn't broken, what changed? Three converging trends created the current threat landscape:

3.1 Identity Became the Perimeter

As organizations adopted cloud-first architectures, the traditional network perimeter dissolved. Identity — specifically, OAuth tokens — became the primary access mechanism. Microsoft reported in its 2025 Digital Defense Report that identity-based attacks are now the #1 initial access vector, surpassing traditional network exploits.

3.2 Phishing-as-a-Service Industrialized MFA Bypass

Platforms like Tycoon 2FA democratized Adversary-in-the-Middle (AiTM) attacks. By mid-2025, Microsoft reported that Tycoon 2FA was responsible for approximately 62% of all phishing attempts blocked by their systems. The platform was eventually dismantled by Europol in a coordinated takedown of 330 domains, but the techniques it popularized persist.

By The Numbers (Microsoft Telemetry, 2025):
  • 146% increase in AiTM phishing incidents year-over-year
  • ~40,000 AiTM incidents detected daily
  • 62% of phishing blocked was Tycoon 2FA (before takedown)
  • 96,000+ distinct victims linked to Tycoon 2FA globally
  • 13 million malicious emails blocked in October 2025 alone

3.3 Nation-State Actors Adopted OAuth Abuse

In February 2025, Microsoft Threat Intelligence published an advisory on Storm-2372, a threat actor assessed with moderate confidence to align with Russian interests. Storm-2372 pioneered a new phishing technique that abuses the OAuth Device Authorization Grant flow — a legitimate protocol feature — to bypass MFA entirely. This marked a shift from exploiting vulnerabilities to exploiting features.

4. Attack Vector 1: Device Code Phishing (Storm-2372)

The Device Authorization Grant (defined in RFC 8628) was designed for devices with limited input capabilities — smart TVs, IoT devices, and CLI tools. The user authenticates on a separate device (like their phone) by entering a short code displayed on the limited device.

How Storm-2372 Weaponized It:

  1. Social Engineering: Attackers establish rapport with targets via WhatsApp, Signal, or Microsoft Teams, often impersonating prominent individuals relevant to the target's organization
  2. Fake Meeting Invite: The attacker sends a phishing email disguised as a Microsoft Teams meeting invitation
  3. Device Code Delivery: The invitation contains an attacker-generated device code and directs the victim to https://microsoft.com/devicelogin — a completely legitimate Microsoft page
  4. Token Capture: When the victim enters the code and authenticates (including MFA), Microsoft issues OAuth tokens to the attacker's application, not the victim's
  5. Persistent Access: The attacker captures both the access token and refresh token, gaining persistent access to the victim's Microsoft 365 account

Why This Is Devastating:

The victim authenticates on a real Microsoft page. There is no fake login form, no suspicious domain, no browser warning. The MFA challenge is satisfied legitimately. Traditional phishing defenses (URL reputation, browser warnings, email link scanning) are completely bypassed because every component of the flow uses legitimate Microsoft infrastructure.

4.1 Evolved Tactic: Primary Refresh Token (PRT) Abuse

In February 2025, Microsoft observed Storm-2372 evolving their technique. After capturing tokens via device code phishing, the attackers began using the Microsoft Authentication Broker client ID to:

  1. Obtain a refresh token with broker capabilities
  2. Register an attacker-controlled device in Entra ID
  3. Acquire a Primary Refresh Token (PRT)
  4. Use the PRT to access organizational resources as a "trusted device"

This technique grants the attacker persistent, device-level trust within the organization — significantly more dangerous than simple token theft.

5. Attack Vector 2: OAuth Redirection Abuse

This technique, disclosed by multiple security researchers in 2025 and confirmed by Microsoft, exploits a fundamental aspect of OAuth's design: error-triggered redirections.

How It Works:

  1. Attackers register a malicious application in an attacker-controlled Entra ID tenant
  2. They configure the app's redirect URI to point to an attacker-owned domain
  3. They craft a phishing URL targeting login.microsoftonline.com/common/oauth2/v2.0/authorize with intentionally malformed parameters (e.g., invalid scopes or prompt=none)
  4. When the authorization server processes the request and encounters an error, it performs a standards-compliant redirect to the registered redirect URI — which is the attacker's domain
  5. The victim is now on the attacker's site, which may serve credential-harvesting pages or malware downloads

Example Malicious OAuth URL (for awareness — DO NOT use)

# This URL originates from a TRUSTED Microsoft domain
# Email filters and browsers will NOT flag it

https://login.microsoftonline.com/common/oauth2/v2.0/authorize
  ?client_id=ATTACKER_APP_ID
  &response_type=code
  &redirect_uri=https://attacker-site.com/callback
  &scope=INVALID_SCOPE_TO_TRIGGER_ERROR
  &prompt=none

# Result: Microsoft's auth server redirects to:
# https://attacker-site.com/callback?error=invalid_scope
# → User is now on the attacker's phishing site
# → The URL they clicked was login.microsoftonline.com
# → All email/browser reputation checks passed

Why traditional defenses fail: The phishing link originates from login.microsoftonline.com or accounts.google.com — domains with perfect reputation scores. Email gateway filters, URL scanners, and browser Safe Browsing all trust these domains implicitly. Microsoft has confirmed these campaigns target government and public-sector organizations.

6. Attack Vector 3: Consent Phishing & Illicit App Grants

Consent phishing takes advantage of OAuth's consent mechanism — the permission dialog users see when granting an application access to their data.

6.1 The Attack

Attackers register an OAuth application that appears legitimate — often impersonating known brands or using names similar to popular internal tools. They then trick users (via phishing or social engineering) into granting this application broad OAuth permissions. Once consent is granted, the application receives OAuth tokens that provide ongoing access to the user's data.

Aspect Legitimate App Malicious App
Publisher Verified publisher badge Unverified or spoofed publisher
Permissions Minimal scopes (e.g., User.Read) Broad scopes (e.g., Mail.ReadWrite, Files.ReadWrite.All)
Access Duration Short-lived, scoped to session Persistent via refresh tokens
MFA Required Yes, at authentication Bypassed — consent is post-authentication
Critical Risk:

Once a user grants consent, the malicious app has persistent access. Changing the user's password does not revoke the OAuth grant. The attacker maintains access until the admin explicitly revokes the application's consent in Entra ID or Google Workspace admin console.

7. Attack Vector 4: Token Persistence & Theft

OAuth tokens, by design, provide access without requiring re-authentication. This property — essential for user experience — creates significant security implications when tokens are stolen.

7.1 Why Token Theft Persists After Password Reset

A common misconception is that resetting a user's password invalidates all active sessions. With OAuth, this is not automatically true:

  • Access tokens remain valid until they expire (typically 60-90 minutes for Entra ID)
  • Refresh tokens can have lifetimes of days to months depending on configuration
  • Password reset does NOT automatically revoke existing refresh tokens in all configurations
  • Attackers who obtain a refresh token can silently generate new access tokens without any user interaction

7.2 Token Theft via Infostealer Malware

Modern infostealer malware (RedLine, Raccoon, Vidar) specifically targets OAuth and session tokens stored in browser profiles. These tokens are extracted alongside cookies and sent to attacker C2 servers, enabling session hijacking without triggering MFA.

Where Tokens Are Stored (Browser Forensics Reference)

# Chrome/Edge token storage locations (for forensic analysis only)
# Windows:
%LOCALAPPDATA%\Google\Chrome\User Data\Default\Local Storage\
%LOCALAPPDATA%\Microsoft\Edge\User Data\Default\Cookies

# macOS:
~/Library/Application Support/Google/Chrome/Default/Cookies
~/Library/Application Support/Microsoft Edge/Default/Cookies

# Infostealers target these files to extract:
# - Session cookies (bypass MFA entirely)
# - OAuth refresh tokens (persistent access)
# - Saved passwords (credential stuffing)

7.3 Shadow Google Accounts

A vulnerability identified by security researchers affects Google OAuth specifically. When employees use "Sign in with Google" for third-party services (Slack, Zoom, Notion), they may create OAuth grants that persist even after the employee is offboarded. If the organization doesn't revoke these OAuth connections, former employees retain access to corporate resources through these "shadow" OAuth sessions. Google has begun automatically deleting unused OAuth clients to address this.

8. Defense: Conditional Access, App Governance & Detection

8.1 Block Device Code Flow (Critical)

The single most impactful mitigation against Storm-2372 and similar attacks is to block the device code authentication flow via Conditional Access in Microsoft Entra ID.

Entra ID — Block Device Code Flow via Conditional Access

// Conditional Access Policy — Block Device Code Flow
// Source: Microsoft Threat Intelligence Advisory (Feb 2025)
{
  "displayName": "Block Device Code Authentication Flow",
  "state": "enabled",
  "conditions": {
    "users": {
      "includeUsers": ["All"]
    },
    "applications": {
      "includeApplications": ["All"]
    },
    "authenticationFlows": {
      "transferMethods": ["deviceCodeFlow"]
    }
  },
  "grantControls": {
    "operator": "OR",
    "builtInControls": ["block"]
  }
}
Implementation Note:

Before blocking device code flow globally, audit your organization for legitimate uses. Common legitimate uses include: conference room displays, Azure CLI on shared terminals, and IoT devices. For these, create an exception policy that allows device code flow only from specific trusted network locations or compliant devices.

8.2 Restrict OAuth App Registrations

PowerShell — Restrict User Consent to Verified Publishers Only

# Restrict user consent to apps from verified publishers only
# Requires: Microsoft.Graph PowerShell module
# Source: Microsoft Entra ID documentation

Connect-MgGraph -Scopes "Policy.ReadWrite.Authorization"

# Set consent policy: users can only consent to apps
# from verified publishers, and only for low-risk permissions
Update-MgPolicyAuthorizationPolicy -DefaultUserRolePermissions @{
    PermissionGrantPoliciesAssigned = @(
        "managePermissionGrantsForSelf.microsoft-user-default-low"
    )
}

# Verify the change
Get-MgPolicyAuthorizationPolicy | 
    Select-Object -ExpandProperty DefaultUserRolePermissions |
    Select-Object PermissionGrantPoliciesAssigned

# Expected output: microsoft-user-default-low
# This means users can only consent to:
# - Apps from verified publishers
# - For permissions classified as "low impact"
# All other consent requests require admin approval

8.3 Revoke Compromised Tokens

If device code phishing or token theft is suspected, immediately revoke all refresh tokens for the affected user:

Emergency Token Revocation (Microsoft Graph API)

# Revoke all refresh tokens for a compromised user
# Source: Microsoft Graph API documentation
# https://learn.microsoft.com/en-us/graph/api/user-revokesigninsessions

# Using Microsoft Graph API
POST https://graph.microsoft.com/v1.0/users/{userId}/revokeSignInSessions

# Using Azure CLI
az rest --method POST \
  --url "https://graph.microsoft.com/v1.0/users/compromised@company.com/revokeSignInSessions"

# Using PowerShell
Revoke-MgUserSignInSession -UserId "compromised@company.com"

# IMPORTANT: After revocation, also:
# 1. Reset the user's password
# 2. Require MFA re-registration
# 3. Review and revoke any suspicious OAuth app consents
# 4. Check for attacker-registered devices in Entra ID

8.4 Audit OAuth App Permissions Regularly

PowerShell — Audit High-Privilege OAuth Apps in Entra ID

# List all OAuth apps with high-privilege permissions
# Source: Microsoft Graph PowerShell documentation

Connect-MgGraph -Scopes "Application.Read.All","DelegatedPermissionGrant.Read.All"

# Find all delegated permission grants (user consents)
$grants = Get-MgOauth2PermissionGrant -All

# Filter for high-risk scopes
$highRiskScopes = @("Mail.ReadWrite", "Mail.Send", "Files.ReadWrite.All", 
                     "Directory.ReadWrite.All", "User.ReadWrite.All")

$riskyGrants = $grants | Where-Object {
    $highRiskScopes | ForEach-Object { $grant.Scope -match $_ }
}

# Output for review
$riskyGrants | ForEach-Object {
    $app = Get-MgServicePrincipal -ServicePrincipalId $_.ClientId
    [PSCustomObject]@{
        AppName     = $app.DisplayName
        Scopes      = $_.Scope
        ConsentType = $_.ConsentType
        UserId      = $_.PrincipalId
    }
} | Format-Table -AutoSize

9. Detection Engineering: KQL & Sigma Rules

9.1 KQL — Detect Device Code Phishing Attempts

Microsoft Sentinel KQL — Device Code Phishing Detection

// Detect suspicious device code authentication patterns
// Source: Adapted from Microsoft Storm-2372 advisory
// Deploy in: Microsoft Sentinel Analytics Rules

SigninLogs
| where TimeGenerated > ago(24h)
| where AuthenticationProtocol == "deviceCode"
| where ResultType == 0  // Successful sign-ins only
| extend DeviceDetail = parse_json(DeviceDetail)
| extend LocationDetail = parse_json(LocationDetails)
| project
    TimeGenerated,
    UserPrincipalName,
    AppDisplayName,
    IPAddress,
    Location = strcat(LocationDetail.city, ", ", 
               LocationDetail.countryOrRegion),
    DeviceOS = DeviceDetail.operatingSystem,
    IsManaged = DeviceDetail.isManaged,
    ConditionalAccessStatus
| where IsManaged != true  // Unmanaged devices are suspicious
| summarize
    AttemptCount = count(),
    DistinctIPs = dcount(IPAddress),
    Apps = make_set(AppDisplayName),
    Locations = make_set(Location)
    by UserPrincipalName, bin(TimeGenerated, 1h)
| where AttemptCount > 1 or DistinctIPs > 1

Sigma Rule: Suspicious OAuth Application Consent

title: Suspicious OAuth Application Consent Grant
id: a2c1b3d4-oauth-consent-2026
status: experimental
description: |
    Detects when a user grants consent to an OAuth application
    with high-privilege permissions from an unverified publisher,
    which may indicate consent phishing.

date: 2026/03/06

    - https://learn.microsoft.com/en-us/entra/identity/
    - https://thehgtech.com/guides/oauth-security-attacks-defense-2026.html

        Category: "ApplicationManagement"

    - Internal line-of-business applications
    - Development/test applications
level: high

    - attack.t1550.001

10. OAuth Security Hardening Checklist

Priority Actions for Security Teams:
Priority Action Mitigates
P0 Block device code flow via Conditional Access Storm-2372, device code phishing
P0 Restrict user consent to verified publishers only Consent phishing, illicit app grants
P0 Deploy phishing-resistant MFA (FIDO2/passkeys) AiTM, token interception
P1 Audit all existing OAuth app permissions quarterly Over-privileged apps, shadow access
P1 Configure token lifetime policies (reduce refresh token lifetime) Token persistence attacks
P1 Enable Continuous Access Evaluation (CAE) Stolen token reuse
P2 Deploy KQL/Sigma detection rules from this guide Device code phishing, consent abuse
P2 Implement app governance policies in Defender for Cloud Apps Unauthorized app access
P2 Revoke OAuth grants for offboarded employees Shadow accounts, persistent access
P2 Train users to recognize device code and consent phishing Social engineering
