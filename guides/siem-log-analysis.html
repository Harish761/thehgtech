SIEM Log Analysis for Beginners

ðŸ“– 12 min read Beginner ðŸ“… Updated: November 2025

Security Information and Event Management (SIEM) platforms are the backbone of modern security operations. They collect, normalize, and analyze logs from across your IT infrastructure to detect security threats. This guide will teach you the fundamentals of SIEM log analysis, from understanding log sources to triaging alerts effectively.

What is a SIEM?

A SIEM is a centralized platform that:

  • Collects logs from firewalls, servers, endpoints, applications, and cloud services
  • Normalizes data into a common format for analysis
  • Correlates events to identify suspicious patterns
  • Generates alerts when security rules are triggered
  • Provides dashboards for real-time monitoring
  • Enables threat hunting through historical log searches
Popular SIEM Platforms: Splunk, IBM QRadar, Microsoft Sentinel, Elastic Security, LogRhythm, ArcSight, Sumo Logic, Chronicle

Common Log Sources

1. Network Devices

  • Firewalls - Connection attempts, blocked traffic, rule hits
  • IDS/IPS - Intrusion attempts, signature matches, anomalies
  • Proxies - Web traffic, URL filtering, user activity
  • VPN - Remote access sessions, authentication events
  • Routers/Switches - Network topology changes, port status

2. Endpoints

  • Windows Event Logs - Logons, process creation, registry changes
  • Linux Syslog - System events, authentication, sudo commands
  • EDR/Antivirus - Malware detections, suspicious behavior
  • Application Logs - Errors, crashes, user actions

3. Identity & Access

  • Active Directory - User/group changes, password resets, lockouts
  • LDAP - Authentication events, directory queries
  • SSO/IAM - Login attempts, MFA events, privilege escalation

4. Cloud Services

  • AWS CloudTrail - API calls, resource changes, access logs
  • Azure Activity Logs - Subscription changes, resource modifications
  • GCP Audit Logs - Admin activity, data access, system events
  • Office 365 - Email activity, file sharing, admin actions

Understanding Log Formats

Syslog Example

Nov 28 10:15:32 webserver01 sshd[1234]: Failed password for admin from 192.168.1.100 port 22 ssh2

Breakdown:

  • Nov 28 10:15:32 - Timestamp
  • webserver01 - Hostname
  • sshd[1234] - Process name and PID
  • Failed password... - Event description

Windows Event Log Example

EventID: 4625
Source: Microsoft-Windows-Security-Auditing
Account Name: john.doe
Failure Reason: Unknown user name or bad password
Workstation Name: DESKTOP-ABC123
Source Network Address: 10.0.0.50

Log Correlation Rules

Correlation rules combine multiple log events to detect complex attack patterns. Here are common examples:

1. Brute Force Attack Detection

# Rule: Detect 5+ failed logins from same IP within 5 minutes
IF (event.type == "failed_login")
AND (count(source_ip) >= 5)
WITHIN (5 minutes)
THEN
    ALERT "Brute Force Attack Detected"
    SEVERITY: HIGH

2. Lateral Movement Detection

# Rule: Detect same user logging into 10+ different systems in 1 hour
IF (event.type == "successful_login")
AND (count(DISTINCT destination_host) >= 10)
BY (username)
WITHIN (1 hour)
THEN
    ALERT "Potential Lateral Movement"
    SEVERITY: CRITICAL

3. Data Exfiltration Detection

# Rule: Detect large outbound data transfer to external IP
IF (network.direction == "outbound")
AND (destination_ip NOT IN internal_networks)
AND (bytes_transferred > 1GB)
THEN
    ALERT "Potential Data Exfiltration"
    SEVERITY: HIGH

Alert Triage Process

Step 1: Validate the Alert

  • Is this a true positive or false positive?
  • What triggered the alert?
  • Is the activity expected or anomalous?

Step 2: Gather Context

  • Who? - Which user or system is involved?
  • What? - What action was performed?
  • When? - When did it occur?
  • Where? - Which systems/networks are affected?
  • Why? - Is there a legitimate business reason?

Step 3: Investigate

  • Search for related events before and after the alert
  • Check if the user/system has a history of similar activity
  • Correlate with threat intelligence (known bad IPs, domains)
  • Review user behavior analytics for anomalies

Step 4: Determine Severity

Severity Criteria Response Time
Critical Active breach, data loss, system compromise Immediate
High Attempted breach, malware detected, privilege escalation < 1 hour
Medium Policy violation, suspicious activity, failed attack < 4 hours
Low Informational, minor policy violation < 24 hours

Step 5: Take Action

  • Escalate - Forward to incident response team if confirmed threat
  • Contain - Isolate affected systems, block malicious IPs
  • Document - Record findings, actions taken, and timeline
  • Close - Mark as false positive or resolved

Common Attack Patterns to Look For

1. Reconnaissance

  • Port scans from external IPs
  • Excessive DNS queries
  • Directory enumeration attempts
  • Vulnerability scanning activity

2. Initial Access

  • Brute force login attempts
  • Phishing email delivery
  • Exploitation of public-facing applications
  • Drive-by downloads

3. Persistence

  • New scheduled tasks created
  • Registry run keys modified
  • New user accounts created
  • Service installations

4. Privilege Escalation

  • Unusual privilege changes
  • Exploitation of vulnerabilities
  • Credential dumping attempts
  • Token manipulation

5. Lateral Movement

  • Pass-the-hash attacks
  • Remote desktop connections
  • PsExec usage
  • WMI/PowerShell remoting

Best Practices

  • Normalize timestamps - Ensure all logs use the same timezone
  • Enrich with context - Add asset info, user details, threat intel
  • Tune rules regularly - Reduce false positives, improve detection
  • Document everything - Keep investigation notes and playbooks
  • Automate responses - Use SOAR for common actions
  • Review dashboards daily - Monitor trends and anomalies
  • Test your rules - Validate detection logic with simulations

Common Mistakes to Avoid

  • Alert fatigue - Too many low-quality alerts overwhelm analysts
  • Ignoring context - Alerts without context are hard to triage
  • Poor log retention - Insufficient historical data limits investigations
  • Lack of documentation - Undocumented rules are hard to maintain
  • No baseline - Without normal behavior, everything looks suspicious

Practical Exercise

Scenario: Suspicious Login Activity

You receive an alert: "Multiple failed logins followed by successful login from unusual location"

Investigation Steps:

  1. Check the user's normal login patterns (location, time, device)
  2. Review the failed login attempts (how many, from where)
  3. Examine the successful login (IP geolocation, user agent)
  4. Search for post-login activity (file access, privilege changes)
  5. Contact the user to verify if login was legitimate
  6. If malicious: Reset password, revoke sessions, investigate further

Conclusion

SIEM log analysis is a critical skill for SOC analysts. By understanding log sources, correlation rules, and alert triage processes, you can effectively detect and respond to security threats.
