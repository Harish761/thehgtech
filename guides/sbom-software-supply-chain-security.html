CycloneDX vs SPDX deep dive, Syft/Trivy SBOM generation, SLSA framework levels, VEX vulnerability triage, CI/CD pipeline integration, and EO 14028/EU CRA compliance

30 min read | Intermediate to Advanced | February 2026
Research Ethics: This document is published for defensive security research and educational purposes. Attack case studies are based on publicly disclosed incidents and post-mortems. Exploit code is presented solely for understanding defensive detection capabilities.
Abstract: The software supply chain is the #1 emerging attack surface in 2026. SolarWinds (2020), Log4Shell (2021), the XZ Utils backdoor (2024), and the Polyfill.io compromise (2024) proved that attackers are shifting from targeting applications directly to compromising their dependencies. A Software Bill of Materials (SBOM) is the foundational defense â€” a machine-readable inventory of every component in your software. This guide covers SBOM generation with Syft, Trivy, and cdxgen; deep comparison of CycloneDX vs SPDX formats; the SLSA framework for build integrity; VEX documents for vulnerability triage; CI/CD pipeline integration patterns; and compliance with US Executive Order 14028 and the EU Cyber Resilience Act.

4. SBOM Generation: Syft, Trivy, cdxgen

Syft â€” Generate SBOM from Source/Image

# Install Syft
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s -- -b /usr/local/bin

# Generate CycloneDX SBOM from project directory
syft dir:./my-app -o cyclonedx-json > sbom.cdx.json

# Generate SPDX SBOM from container image
syft registry:myorg/api:v3.2.1 -o spdx-json > sbom.spdx.json

# Generate from Dockerfile (build-time analysis)
syft docker:myorg/api:latest -o cyclonedx-json > sbom.cdx.json

# Include file metadata and secrets detection
syft dir:./my-app -o cyclonedx-json \
  --catalogers all \
  --source-name "my-api-service" \
  --source-version "3.2.1" > sbom.cdx.json

Trivy â€” SBOM + Vulnerability Scan in One

# Install Trivy
curl -sfL https://raw.githubusercontent.com/aquasecurity/trivy/main/contrib/install.sh | sh -s -- -b /usr/local/bin

# Generate SBOM and scan for vulnerabilities simultaneously
trivy image --format cyclonedx --output sbom.cdx.json myorg/api:v3.2.1

# Scan an existing SBOM for vulnerabilities
trivy sbom sbom.cdx.json

# Generate SBOM from filesystem with license detection
trivy fs --format spdx-json --output sbom.spdx.json \
  --scanners vuln,license ./my-app

# Scan and FAIL if critical vulns found (CI/CD gate)
trivy sbom sbom.cdx.json --exit-code 1 --severity CRITICAL,HIGH

cdxgen â€” Language-Aware CycloneDX Generator

# Install cdxgen (supports 40+ languages)
npm install -g @cyclonedx/cdxgen

# Generate SBOM (auto-detects language/framework)
cdxgen -o sbom.cdx.json ./my-app

# Generate with vulnerability data embedded
cdxgen -o sbom.cdx.json --deep ./my-app

# Generate for specific project type
cdxgen -t python -o sbom.cdx.json ./my-python-app
cdxgen -t java -o sbom.cdx.json ./my-java-app

5. SLSA Framework: Build Integrity Levels

SLSA (Supply-chain Levels for Software Artifacts) answers the critical question: "Was this binary actually built from the source code I audited, by the build system I trust?"

LevelRequirementsProtects Against
SLSA 1Build provenance exists (automated build, provenance document)Accidental errors, basic tampering awareness
SLSA 2Signed provenance from a hosted build service (GitHub Actions, Cloud Build)Tampering after build, unsigned artifacts
SLSA 3Hardened CI, non-falsifiable provenance, source-defined buildCompromised CI config, insider threats in pipeline
SLSA 4Hermetic/reproducible builds, full dependency provenanceNation-state supply chain attacks (SolarWinds-class)

GitHub Actions â€” SLSA Level 3 Provenance

# .github/workflows/slsa-build.yml
name: SLSA Build with Provenance
on:
  push:
    tags: ['v*']

permissions:
  id-token: write    # For signing provenance
  contents: read
  attestations: write

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      digest: ${{ steps.build.outputs.digest }}
    steps:
      - uses: actions/checkout@v4

      - name: Build artifact
        id: build
        run: |
          npm ci --ignore-scripts  # Deterministic install
          npm run build
          DIGEST=$(sha256sum dist/app.tar.gz | cut -d' ' -f1)
          echo "digest=sha256:${DIGEST}" >> "$GITHUB_OUTPUT"

      - name: Generate SBOM
        run: |
          syft dir:. -o cyclonedx-json > sbom.cdx.json

      # SLSA Provenance â€” creates signed, non-falsifiable build attestation
      - name: Attest Build Provenance
        uses: actions/attest-build-provenance@v2
        with:
          subject-name: 'myorg/api'
          subject-digest: ${{ steps.build.outputs.digest }}
          push-to-registry: true

      # SBOM Attestation â€” links SBOM to the artifact
      - name: Attest SBOM
        uses: actions/attest-sbom@v1
        with:
          subject-name: 'myorg/api'
          subject-digest: ${{ steps.build.outputs.digest }}
          sbom-path: 'sbom.cdx.json'

6. VEX: Vulnerability Exploitability Exchange

An SBOM tells you what components you have. A VEX document tells you whether the vulnerability in that component actually affects your product. This solves the "alert fatigue" problem â€” not every CVE in a dependency is exploitable in your context.

VEX StatusMeaningExample
Not AffectedVulnerability exists in component but is not exploitable in this productLog4Shell in log4j used only for compile-time annotation processing

FixedWas affected, now patchedUpdated lodash from 4.17.20 to 4.17.21
Under InvestigationStill determining exploitabilityNew CVE disclosed, analysis in progress

Complete CI Pipeline with SBOM Gate

# .github/workflows/sbom-pipeline.yml
name: SBOM Security Pipeline
on: [push, pull_request]

jobs:
  sbom-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build Application
        run: npm ci && npm run build

      # Step 1: Generate SBOM
      - name: Generate SBOM (CycloneDX)
        run: |
          curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s
          syft dir:. -o cyclonedx-json > sbom.cdx.json
          echo "Components found: $(jq '.components | length' sbom.cdx.json)"

      # Step 2: Vulnerability scan with gate
      - name: Scan SBOM for Vulnerabilities
        run: |
          curl -sfL https://raw.githubusercontent.com/aquasecurity/trivy/main/contrib/install.sh | sh -s
          trivy sbom sbom.cdx.json \
            --exit-code 1 \
            --severity CRITICAL \
            --format json \
            --output vuln-report.json
        continue-on-error: false  # FAIL the build on CRITICAL

      # Step 3: License compliance check
      - name: Check Licenses
        run: |
          # Extract all licenses from SBOM
          jq -r '.components[].licenses[]?.license.id // "UNKNOWN"' \
            sbom.cdx.json | sort | uniq -c | sort -rn
          
          # Fail on copyleft licenses in proprietary project
          BANNED=$(jq -r '.components[] | select(.licenses[]?.license.id == "GPL-3.0-only") | .name' sbom.cdx.json)
          if [ -n "$BANNED" ]; then
            echo "âŒ GPL-3.0 components found: $BANNED"
            exit 1
          fi

      # Step 4: Upload to Dependency-Track
      - name: Upload SBOM to Dependency-Track
        run: |
          curl -X POST "https://dtrack.internal.company.com/api/v1/bom" \
            -H "X-Api-Key: ${{ secrets.DTRACK_API_KEY }}" \
            -H "Content-Type: multipart/form-data" \
            -F "autoCreate=true" \
            -F "projectName=my-api-service" \
            -F "projectVersion=${{ github.sha }}" \
            -F "bom=@sbom.cdx.json"

      # Step 5: Store SBOM as build artifact
      - uses: actions/upload-artifact@v4
        with:
          name: sbom
          path: sbom.cdx.json

8. Attack Case Studies & Detection

8.1 XZ Utils Backdoor (CVE-2024-3094) â€” The Social Engineering Supply Chain Attack

The Attack: Over 2+ years, a pseudonymous contributor ("Jia Tan") systematically gained trust of the sole XZ Utils maintainer through helpful contributions. After becoming a co-maintainer, they injected an obfuscated backdoor into the build system (not the source code) that would execute during compilation on Debian/RPM-based systems. The backdoor hijacked the SSH authentication flow via systemd's dependency on liblzma, allowing the attacker to execute arbitrary commands before authentication.

Why SBOMs Help: An SBOM tracking build-time dependencies and provenance would have flagged that the liblzma binary did not match a clean build from the audited source code â€” a SLSA Level 3 violation.

Detection: Compare binary hash in SBOM against reproducible build output. Mismatch = compromised build pipeline.

8.2 Detection: YARA Rule for Malicious NPM Packages

YARA Rule: Suspicious NPM Install Scripts

rule npm_supply_chain_indicators {
    meta:
        description = "Detects suspicious patterns in NPM packages"
        author = "TheHGTech"
        date = "2026-02-18"
        severity = "high"
    
    strings:
        // Obfuscated eval patterns
        $eval1 = /eval\s*\(\s*Buffer\.from\s*\(/ nocase
        $eval2 = /eval\s*\(\s*atob\s*\(/ nocase
        
        // DNS/HTTP exfiltration in install scripts
        $exfil1 = /https?:\/\/[a-z0-9]{8,}\./ nocase
        $exfil2 = /child_process.*exec.*curl/ nocase
        $exfil3 = /child_process.*exec.*wget/ nocase
        
        // Environment variable harvesting
        $env1 = /process\.env\.(AWS_|GITHUB_TOKEN|NPM_TOKEN)/ nocase
        $env2 = /os\.environ\[.(AWS_|GITHUB|CI_)/ nocase
        
        // Suspicious install script targets
        $install1 = "preinstall" 
        $install2 = "postinstall"
    
    condition:
        any of ($eval*) or
        (any of ($exfil*) and any of ($install*)) or
        (any of ($env*) and any of ($exfil*))
}
