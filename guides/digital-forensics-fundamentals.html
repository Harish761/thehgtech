Digital Forensics Fundamentals

DFIR

Master evidence collection, chain of custody, and analysis techniques for incident investigations.

24 min read
Intermediate to Advanced
Jan 17, 2026

When a security incident occurs, the clock is ticking. Evidence is volatile - memory is ephemeral, logs rotate, and attackers cover their tracks. Digital forensics provides the methodology to properly collect, preserve, and analyze this evidence before it's gone.

What is DFIR?

Digital Forensics and Incident Response (DFIR) combines forensic analysis with incident handling. Forensics focuses on evidence and investigation; IR focuses on containment, eradication, and recovery. Together, they answer: What happened? How did it happen? What's the impact?

1. Forensic Principles

The Four A's of Forensics

  1. Acquire - Collect evidence without altering it
  2. Authenticate - Prove evidence hasn't been tampered with (hashes)
  3. Analyze - Examine evidence for relevant findings
  4. Archive - Preserve evidence for future reference/legal needs

Key Principles

  • Never work on original evidence - Always create forensic copies
  • Document everything - Screenshots, photos, detailed notes
  • Maintain integrity - Use write blockers, verify hashes
  • Be methodical - Follow established procedures
  • Preserve context - Timestamps, system state, who was logged in

2. Chain of Custody

Chain of custody is critical for legal proceedings. It documents every hand that touches the evidence.

Breaking Chain = Evidence Inadmissible

If chain of custody is broken or poorly documented, evidence may be thrown out in court. Even for internal investigations, proper documentation protects against claims of tampering.

Chain of Custody Form Elements

Required Documentation
  • Case Information: Case number, incident date, description
  • Evidence Description: Device type, serial number, model, condition
  • Collection Details: Who, when, where, how
  • Hash Values: MD5 and SHA256 of original evidence
  • Transfer Log: Every handoff - from/to, date/time, purpose, signatures
  • Storage Location: Evidence locker, safe, restricted access area

3. Order of Volatility

Collect evidence in order of volatility - most volatile (easiest to lose) first:

Priority Evidence Type Volatility Window
1 CPU registers, cache Nanoseconds Gone immediately
2 RAM (memory) Seconds Lost on reboot
3 Network connections Minutes Sessions close
4 Running processes Minutes Change constantly
5 Disk (HDD/SSD) Days-Months Persistent until overwritten
6 Backups, archives Years Retention policies
7 Printouts, media Years Physical storage
Collection Order
  1. Photograph the scene and screen
  2. Capture memory dump
  3. Document running processes and network connections
  4. Create forensic disk image
  5. Collect relevant logs

4. Memory Forensics

Memory contains evidence that doesn't exist on disk: running malware, decrypted data, network connections, and command history.

Memory Acquisition

# Windows - Using WinPMEM
winpmem_mini_x64.exe memory.raw

# Linux - Using LiME
insmod lime.ko "path=/tmp/memory.lime format=lime"

# Using FTK Imager (GUI)
File → Capture Memory → Select destination

Memory Analysis with Volatility 3

# Identify operating system profile
vol -f memory.raw windows.info

# List running processes
vol -f memory.raw windows.pslist

# Process tree (parent-child relationships)
vol -f memory.raw windows.pstree

# Network connections
vol -f memory.raw windows.netstat

# Command line arguments
vol -f memory.raw windows.cmdline

# Detect hidden/injected code
vol -f memory.raw windows.malfind

# Extract password hashes
vol -f memory.raw windows.hashdump

What to Look For in Memory

  • Unusual parent processes - CMD spawned from Word?
  • Process injection - Legitimate process with injected code
  • Orphaned processes - No parent (parent killed/crashed)
  • Suspicious network connections - Strange IPs/ports
  • Encoded commands - Base64, XOR in command lines

5. Disk Forensics

Creating Forensic Images

Always Use Write Blockers

Hardware or software write blockers prevent accidental modification of evidence. Never connect suspect drives directly to your forensic workstation without protection.

# Linux - Using dd (basic)
dd if=/dev/sda of=/evidence/disk.img bs=4M status=progress

# Linux - Using dc3dd (better logging)
dc3dd if=/dev/sda of=/evidence/disk.img hash=md5 hash=sha256 log=/evidence/dd.log

# Verify the image
md5sum /dev/sda
md5sum /evidence/disk.img
# Hashes must match!

Mounting Images Read-Only

# Mount forensic image read-only
mount -o ro,loop,noexec /evidence/disk.img /mnt/evidence

# Mount E01 images with ewfmount
ewfmount /evidence/image.E01 /mnt/ewf
mount -o ro /mnt/ewf/ewf1 /mnt/evidence

6. Windows Artifacts

Key Locations

Artifact Location Use Case
Event Logs C:\Windows\System32\winevt\Logs\ Security, application, system events
Registry Hives C:\Windows\System32\config\ System configuration, user activity
Prefetch C:\Windows\Prefetch\ Program execution evidence
Amcache C:\Windows\AppCompat\Programs\ Program execution history
SRUM C:\Windows\System32\sru\ Application usage, network activity
Browser Data %APPDATA%\Local\ History, downloads, cookies
$MFT NTFS Master File Table All file metadata, deleted files
$USNJrnl NTFS Change Journal File system changes over time

Eric Zimmerman's Tools

Essential free Windows forensics tools:

  • MFTECmd - Parse $MFT for file timeline
  • PECmd - Parse Prefetch files
  • EvtxECmd - Parse Windows Event Logs
  • RECmd - Registry analysis
  • AmcacheParser - Amcache analysis
  • Timeline Explorer - View parsed data
# Parse Prefetch for execution history
PECmd.exe -d C:\Windows\Prefetch --csv output\

# Parse MFT for file timeline
MFTECmd.exe -f $MFT --csv output\

# Parse Security Event Log
EvtxECmd.exe -f Security.evtx --csv output\

7. Linux Artifacts

Key Locations

Artifact Location Use Case
Auth logs /var/log/auth.log or /var/log/secure SSH, sudo, authentication
Syslog /var/log/syslog or /var/log/messages System events
Bash history ~/.bash_history Command history
Cron jobs /etc/crontab, /var/spool/cron/ Scheduled tasks
User accounts /etc/passwd, /etc/shadow Account enumeration
SSH keys ~/.ssh/ Authorized access
Web logs /var/log/apache2/, /var/log/nginx/ Web attacks
Systemd journal /var/log/journal/ Service logs

Useful Commands

# Find recently modified files
find / -mtime -1 -type f 2>/dev/null

# Find SUID binaries (privilege escalation)
find / -perm -4000 -type f 2>/dev/null

# Check cron for persistence
cat /etc/crontab
ls -la /etc/cron.*/

# Check for unusual users
cat /etc/passwd | awk -F: '$3 >= 1000 {print}'

# Review SSH authentication
grep "Accepted\|Failed" /var/log/auth.log

9. Documentation and Reporting

Report Structure

  1. Executive Summary - Non-technical overview, key findings, impact
  2. Scope and Methodology - What was examined and how
  3. Timeline of Events - Chronological sequence of actions
  4. Detailed Findings - Evidence with screenshots and analysis
  5. Indicators of Compromise - Hashes, IPs, domains, filenames
  6. Conclusions - What happened, attribution if possible
  7. Recommendations - Remediation and prevention steps
  8. Appendices - Full logs, tool output, chain of custody
Next Steps
