Ransomware Response Playbook 2026

Complete incident response guide for ransomware attacks: containment, negotiation decisions, recovery, and hardening

25 min read | Intermediate | January 2026
Time is Critical: The first 60 minutes after ransomware detection determine your recovery success. Have this playbook ready BEFORE an incident occurs. Print it, distribute it, and practice it.

1. Understanding Modern Ransomware

Modern ransomware is not simple malware. Today's attacks are sophisticated, multi-stage operations run by organized criminal enterprises with help desks, negotiators, and affiliate programs.

The Evolution of Ransomware

Era Characteristics Example
2013-2016 Spray-and-pray, low ransoms CryptoLocker
2017-2019 Targeted attacks, higher ransoms WannaCry, NotPetya
2020-2022 Double extortion (encrypt + leak) Maze, REvil
2023-2024 Triple extortion (+ DDoS/customer targeting) LockBit, BlackCat
2025-2026 RaaS maturity, AI-assisted attacks RansomHub, Akira

Common Attack Vectors in 2026

Vector Percentage Prevention
Phishing/Social Engineering 36% Security awareness, email filtering
Exploited Vulnerabilities 29% Patch management, vuln scanning
Stolen Credentials 24% MFA, credential monitoring
RDP/VPN Compromise 8% ZTNA, MFA, monitoring
Supply Chain 3% Vendor assessment, SBOM
Think of it like this: Modern ransomware groups operate like legitimate businesses. They have customer service, payment portals, and even satisfaction guarantees. This professionalization makes them more dangerous and more predictable.

2. Detection and Initial Response

Signs of Active Ransomware

  • Ransom notes appearing on desktops or in folders
  • File extensions changing to .encrypted, .locked, .crypted, etc.
  • Unusual CPU/disk activity during off-hours
  • Security tools disabled or alerting on tampering
  • Network connections to unusual IPs/domains
  • Shadow copies deleted (vssadmin commands)

First 15 Minutes Checklist

  • Confirm it is ransomware (not a drill, not a prank)
  • DO NOT shut down affected systems (preserve memory evidence)
  • Disconnect affected systems from network (pull cables, disable WiFi)
  • Take photos of ransom notes and any visible IOCs
  • Activate incident response team
  • Start incident documentation log
  • Notify CISO/IT leadership
  • Secure backup systems (take offline if not already)
Critical Warning: DO NOT attempt to decrypt files yourself, run anti-malware scans, or engage with attackers without proper preparation. Each of these actions can destroy evidence or worsen your position.

3. Containment Phase

The goal of containment is to stop the bleeding without destroying evidence.

Network Containment Actions

Immediate Network Actions

  1. Isolate affected VLANs - Block lateral movement at the switch level
  2. Disable compromised accounts - Reset passwords for affected users
  3. Block C2 domains/IPs - Update firewall rules immediately
  4. Disconnect WAN links if multiple sites affected
  5. Preserve DNS logs - Critical for identifying scope

Endpoint Containment

# Windows - Isolate endpoint while preserving evidence
# Option 1: Block all network except management
netsh advfirewall set allprofiles firewallpolicy blockinbound,blockoutbound

# Option 2: Use EDR to quarantine
# CrowdStrike: Contain host from console
# Defender: Isolate device from portal

# Collect volatile evidence FIRST
winpmem.exe memory.raw  # Memory dump
wmic process list full > processes.txt
netstat -ano > connections.txt

What NOT to Do During Containment

Avoid This Why It Is Dangerous
Shutting down systems Destroys memory evidence, may trigger deadman switches
Running AV scans immediately May delete ransomware before analysis, alter timestamps
Restoring from backups immediately May restore infected files, wastes time if scope unknown
Contacting attackers without preparation Shows desperation, weakens negotiation position

4. Impact Assessment

Before deciding on recovery strategy, you must understand the full scope of the attack.

Scope Assessment Checklist

Question How to Determine Impact on Response
How many systems affected? EDR console, manual survey Scale of recovery effort
What data was encrypted? File server scans, share analysis Business criticality
Was data exfiltrated? Firewall logs, DLP alerts Double extortion risk
How long were they inside? SIEM analysis, forensics Backup viability
Are backups clean? Isolated backup testing Recovery feasibility
What ransomware variant? Ransom note, file extension Decryptor availability

Identify the Ransomware

Knowing the variant helps determine:

  • Whether free decryptors exist (check NoMoreRansom.org)
  • The group's reputation for providing working decryptors
  • Typical ransom amounts and negotiation patterns
  • Known TTPs to help identify scope
Resource: ID Ransomware (id-ransomware.malwarehunterteam.com) can identify the variant from a ransom note or encrypted file sample.

5. The Ransom Decision Framework

The question everyone dreads: should you pay? There is no universal answer, but this framework helps structure the decision.

Legal Notice: Paying ransoms to sanctioned entities is illegal (OFAC). Always involve legal counsel and check the group against sanctions lists before any payment consideration.

Factors Against Paying

Factor Consideration
Viable backups exist Recovery without payment is possible
Decryptor available NoMoreRansom.org has free tool
Group unreliable Known for not providing keys
Sanctions risk Group on OFAC list
Insurance prohibits Policy does not cover ransom

Factors That May Support Paying

Factor Consideration
Life-safety data Healthcare, infrastructure critical data
No viable backups Rebuild would be catastrophic
Business continuity threat Company survival at stake
Data exposure risk Sensitive data will be leaked

If You Decide to Negotiate

  1. Engage professional negotiators - Many cyber insurance policies include this
  2. Buy time - Claim complexity, ask for extensions
  3. Verify capability - Request proof of decryption for small file
  4. Negotiate price - Initial demands are often negotiable by 50-70%
  5. Use cryptocurrency safely - Work with reputable exchange
  6. Get everything in writing - Screen captures of all communications

6. Recovery and Restoration

Recovery Priority Framework

Tier 1 - Critical (0-4 hours)

  • Domain Controllers
  • DNS/DHCP servers
  • Authentication systems
  • Core network infrastructure

Tier 2 - Essential (4-24 hours)

  • Email systems
  • Primary business applications
  • Customer-facing systems
  • Database servers

Tier 3 - Important (24-72 hours)

  • File servers
  • Development environments
  • Secondary applications
  • User workstations

Clean Rebuild vs. Restore

Approach When to Use Pros/Cons
Clean Rebuild Widespread infection, backdoors suspected Eliminates persistence
Time-consuming
Restore from Backup Clean backup verified, limited scope Faster recovery
May restore malware
Hybrid Approach Most common - rebuild OS, restore data Balance of speed/security

Validation Before Reconnection

Before bringing systems back online:

  • All indicators of compromise (IOCs) scanned and cleared
  • All credentials reset (especially service accounts)
  • Security patches applied to rebuilt systems
  • EDR/AV fully operational and updated
  • MFA enabled on all accounts
  • Monitoring/alerting configured

7. Communication and Notification

Internal Communication Timeline

When Audience Message
Immediately IR Team, CISO Incident declared, containment underway
Within 1 hour Executive team, Legal Situation brief, business impact
Within 4 hours All employees What is happening, what to do/not do
Ongoing Board (if significant) Status updates per governance requirements

External Notification Requirements

Regulation Deadline Trigger
GDPR (EU) 72 hours Personal data breach
SEC (US Public Co.) 4 business days Material cybersecurity incident
HIPAA (Healthcare) 60 days PHI breach affecting 500+
State Laws (varies) 24-72 hours Resident data exposed
CIRCIA (Critical Infra) 72 hours Significant cyber incident
Tip: Pre-draft notification templates as part of your IR planning. Under pressure, you will not write well. Legal should review these templates annually.

8. Post-Incident Hardening

After recovery, do not just return to normal. Use this crisis to drive security improvements the organization might otherwise resist.

Immediate Hardening (Week 1-2)

  • Reset ALL passwords organization-wide (including service accounts)
  • Enable MFA on all accounts (no exceptions)
  • Review and restrict administrative privileges
  • Implement application whitelisting on critical servers
  • Update and test backup procedures
  • Deploy enhanced endpoint protection

Medium-Term Improvements (Month 1-3)

  • Network segmentation - Limit lateral movement capability
  • Zero Trust implementation - Verify before trusting any connection
  • Enhanced email security - Advanced phishing protection
  • Privileged Access Management - Vault and rotate sensitive credentials
  • 24/7 Security Operations - MDR or internal SOC coverage

Lessons Learned Process

Within 2 weeks of recovery, conduct a formal post-incident review:

Question Purpose
How did attackers get in? Fix the initial access vector
Why was it not detected earlier? Improve detection capabilities
What slowed our response? Improve response processes
What worked well? Reinforce successful practices
What do we need that we did not have? Budget justification
Key Takeaway: A ransomware incident is traumatic, but organizations that treat it as a learning opportunity emerge stronger. Use the crisis to drive the security investments that were previously too expensive.
