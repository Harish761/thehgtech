What is Clawdbot/Moltbot?

Clawdbot (now renamed Moltbot) is an open-source, self-hosted AI assistant that functions like a personal JARVIS for your digital life. Created by Peter Steinberger, it became one of the fastest-growing GitHub repositories ever, amassing nearly 30,000 stars within weeks of its release.

Unlike cloud-based AI assistants like ChatGPT or Claude, Clawdbot runs entirely on your own infrastructure, giving you complete control over your data - but also complete responsibility for security.

Key Distinction: Clawdbot is NOT the same as "ClaudeBot" (Anthropic's web crawler). Clawdbot is a personal AI assistant that was renamed to Moltbot after Anthropic raised trademark concerns.

Core Concept

Think of Clawdbot as an AI-powered command center that:

  • Runs locally on your machine or server
  • Integrates with your messaging apps (WhatsApp, Telegram, Slack, Discord, iMessage, Teams)
  • Executes shell commands on your behalf
  • Manages files and automates tasks
  • Controls your browser and web interactions
  • Sends proactive notifications and reminders

Critical Security Risks

The combination of extensive system access and rapid adoption without security review has created a perfect storm of vulnerabilities. Here are the critical risks every Clawdbot user must understand:

Exposed Administrative Ports
CRITICAL

Hundreds of Moltbot instances have been found publicly accessible on the internet, many with no authentication whatsoever. This grants attackers full shell access with the privileges of the running user - often root.

Root Cause: Misconfigured reverse proxies treating external connections as internal, or users binding to 0.0.0.0 without understanding the implications.

Plaintext Secrets Storage
CRITICAL

Moltbot stores sensitive information including API keys, SSH keys, AWS credentials, and conversation histories in plaintext Markdown and JSON files on the local filesystem.

Risk: Infostealer malware like RedLine, Lumma, and Vidar specifically target these locations, making Clawdbot users prime targets for credential theft.

Prompt Injection Attacks
CRITICAL

Clawdbot processes external data from emails, web content, and messages - making it highly susceptible to prompt injection. Malicious instructions embedded in untrusted inputs can manipulate the AI into:

  • Bypassing safety protocols
  • Exfiltrating sensitive data
  • Executing unauthorized shell commands
  • Deleting files or wiping systems
Supply Chain Attacks via ClawdHub
HIGH

The ClawdHub skill library allows third-party extensions. Security researchers demonstrated that malicious "skills" can be uploaded that enable:

  • Remote Command Execution (RCE)
  • Data exfiltration to attacker-controlled servers
  • Persistent backdoor installation
Malicious Impersonation
HIGH

A fake VS Code extension named "ClawdBot Agent - AI Coding Assistant" was discovered on the official Microsoft marketplace. This extension silently installed ScreenConnect malware for persistent remote access.

Important: Moltbot does NOT have an official VS Code extension. Any extension claiming to be Clawdbot/Moltbot is likely malicious.
Excessive Permission Requirements
MEDIUM

By design, Clawdbot requires extensive system access: reading private messages, storing auth tokens, executing commands, and maintaining state. This creates a massive attack surface where any compromise leads to total system control.

Real-World Attacks Documented

These aren't theoretical risks - actual attacks on Clawdbot installations have been documented:

Documented Incidents:
  • Failed login attempts - Exposed instances showing brute-force attack logs
  • Credential theft - API keys and tokens stolen from exposed configurations
  • Email inbox wipes - Prompt injection attacks that deleted entire email inboxes
  • Unauthorized shell access - Attackers gained root access on unprotected instances
  • Keychain access requests - macOS users reporting suspicious Keychain prompts after installation

Hardening Guide

If you're determined to use Clawdbot/Moltbot, implement these security measures as an absolute minimum:

1

Run on Isolated Infrastructure

Never run Clawdbot on your primary workstation. Use a dedicated, disposable machine:

  • Cheap VPS from a cloud provider
  • Old computer specifically for this purpose
  • Docker container with limited capabilities

Why: If compromised, only this isolated system is affected - not your main machine with SSH keys, credentials, and sensitive data.

2

Never Expose to Public Internet

The Clawdbot gateway should NEVER be accessible from the public internet. Options:

  • VPN Only: Use Tailscale, WireGuard, or similar to access
  • Localhost Only: Bind to 127.0.0.1, access via SSH tunnel
  • Firewall: Block all incoming connections on gateway ports
3

Enable Strong Authentication

Configure mandatory authentication for all Clawdbot services:

# In your Clawdbot config
gateway:
  auth:
    enabled: true
    method: token
    token: "$(openssl rand -hex 32)"
4

Implement Least Privilege

Don't run as root. Create a dedicated low-privilege user:

# Create dedicated user
sudo useradd -r -s /bin/false moltbot

# Run as that user
sudo -u moltbot moltbot start
5

Encrypt Secrets at Rest

Don't let sensitive data sit in plaintext. Options:

  • Use encrypted filesystem (LUKS, FileVault)
  • Integrate with secret manager (HashiCorp Vault, 1Password CLI)
  • Enable Clawdbot's built-in encryption if available
6

Disable or Vet Skills/Plugins

ClawdHub is a supply chain risk. Either:

  • Disable entirely: Don't use third-party skills
  • Manual vetting: Review source code before installation
  • Allowlist: Only permit specific verified skills
7

Configure Messaging Security

For messaging integrations, reduce attack surface:

# Require explicit @mention in group chats
messaging:
  require_mention: true
  
# Block unknown senders until approved
  dm_pairing: true
  block_unknown: true
8

Run Security Audit

Use Clawdbot's built-in security audit command regularly:

moltbot security audit

This checks for common misconfigurations and exposed services.

Secure Configuration Examples

Firewall Rules (UFW)

# Default deny incoming
sudo ufw default deny incoming
sudo ufw default allow outgoing

# Allow only SSH from your IP
sudo ufw allow from YOUR_IP to any port 22

# DO NOT allow Moltbot ports publicly
# sudo ufw allow 8080  # NEVER DO THIS

# Enable firewall
sudo ufw enable

Secure SSH Configuration

# /etc/ssh/sshd_config

# Disable password auth
PasswordAuthentication no

# Disable root login
PermitRootLogin no

# Only allow specific user
AllowUsers moltbot-admin

Docker Isolation

# docker-compose.yml - Run Moltbot isolated
version: '3.8'
services:
  moltbot:
    image: moltbot/moltbot:latest
    # Bind ONLY to localhost
    ports:
      - "127.0.0.1:8080:8080"
    # Drop all capabilities, add only what's needed
    cap_drop:
      - ALL
    # Read-only filesystem where possible
    read_only: true
    # Limit resources
    deploy:
      resources:
        limits:
          cpus: '1'
          memory: 2G

Tailscale VPN Access

# Install Tailscale on your server
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up

# Now access Moltbot only via Tailscale IP
# e.g., http://100.x.x.x:8080 (never public IP)

Monitoring & Detection

Even with hardening, continuous monitoring is essential to detect compromise attempts or successful breaches.

Log Monitoring

# Watch for authentication failures
tail -f /var/log/moltbot/access.log | grep -i "auth\|fail\|denied"

# Monitor for suspicious commands
tail -f /var/log/moltbot/commands.log | grep -iE "curl|wget|nc|bash|sh|python"

Fail2ban Configuration

# /etc/fail2ban/jail.local
[moltbot]
enabled = true
port = 8080
filter = moltbot
logpath = /var/log/moltbot/access.log
maxretry = 3
bantime = 3600

Network Monitoring

# Check for unexpected outbound connections
ss -tunaep | grep moltbot

# Monitor DNS queries (data exfil detection)
tcpdump -i any port 53 -n
Best Practice: Set up alerting for any authentication failures, unexpected outbound connections, or unusual command patterns. Use SIEM integration if available.

Frequently Asked Questions

Should I use Clawdbot/Moltbot?

That depends on your threat model. If you're a security professional who can properly isolate and monitor it - proceed with caution. If you're a casual user looking for a "cool AI assistant" - the risks likely outweigh the benefits.

Is Moltbot safer than Clawdbot?

No. Moltbot is just a rename due to trademark issues. The code, architecture, and risks are identical.

Can I use it safely on my main computer?

We strongly recommend against this. Use isolated, disposable infrastructure only.

What's the difference between Clawdbot and ClaudeBot?

Completely different things:

  • Clawdbot/Moltbot: Open-source personal AI assistant you self-host
  • ClaudeBot: Anthropic's web crawler that indexes websites

I found an exposed Clawdbot instance. What do I do?

Report it responsibly. Do not access, exploit, or exfiltrate data. Consider reporting to the hosting provider or security research organizations.

Conclusion

Clawdbot/Moltbot represents both the promise and peril of powerful AI assistants. Its capabilities are impressive - but with great power comes great responsibility for security.

Key takeaways:

  • Never expose to the public internet
  • Never run on your primary workstation
  • Never run as root
  • Never trust third-party skills without vetting
  • Always use VPN/private network access
  • Always enable strong authentication
  • Always encrypt secrets at rest
  • Always monitor for suspicious activity

Stay safe out there.
